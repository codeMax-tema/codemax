# CodeMax 可上线竞争级 MVP 最终需求与并行开发文档

## 0. 文档定位

这份文档是给 4 位高级工程师直接并行开发使用的最终需求文档呀。  

因此，本文档不再使用“Demo”“后续再说”“V1 必做 / V2 增强”这类口径。  
凡是写入本文档的能力，均视为本次上线 MVP 的组成部分哦。

---

## 1. 最终产品目标

### 1.1 一句话目标

CodeMax 要交付为：

`以 Codex 式多轮工具调用主链为默认执行方式的桌面端编程智能体产品：模型根据真实工具结果自主搜索、读取、编辑、运行、修复和交付；Runtime 对每次副作用实施安全、隐私、预算、审批和审计控制。`

### 1.2 竞争目标

本次上线 MVP 必须满足：

1. Codex 有的核心任务链能力，CodeMax 必须具备。
2. Cursor 有的常见代码 Agent 使用体验，CodeMax 必须覆盖核心链路。
3. CodeMax 必须在隐私保护、个人画像、运行契约、证据交付上形成明确差异。
4. 用户安装后可以对真实本地仓库创建任务并完成完整闭环。
5. 所有默认路径必须是真实数据，不能依赖 Demo、Fixture、硬编码任务。
6. 编程模式默认主链必须是模型可根据每次真实工具结果改变下一步动作的多轮工具循环，不能是固定的 `TodoPlan -> EditingPlan -> Validation` 节点序列。

### 1.3 产品核心表达

对内统一口径：

`CodeMax 不是另一个会写代码的 Agent，而是一个更懂用户、更保护隐私、更可控、更能审计交付结果的本地编程智能体工作台。`

### 1.4 最终交付主链定义

最终上线版本必须以以下链路完成真实编程任务：

```text
用户目标与运行契约
-> 模型选择下一步受控工具
-> Runtime 校验路径、参数、权限、预算和隐私
-> Runtime 执行或请求审批
-> 真实工具结果、证据引用和截断说明回灌模型
-> 模型根据结果继续搜索 / 读取 / 编辑 / 运行 / 修复 / 审查
-> 交付、等待审批、人工介入、失败或取消
```

这条链路的判定标准如下：

1. 模型不是一次性输出完整编辑计划后由固定状态机照单执行，而是每一轮根据上一轮真实结果自主选择下一步。
2. Runtime 必须以严格 Schema 注册和验证工具；未知工具、非法参数、越权路径、未经审批的高风险操作和预算超限必须被拒绝或暂停。
3. 文件读写、命令、Git、审批、任务状态和最终交付均通过工具调用进入统一审计链；模型不得伪造工具执行结果。
4. 每个工具调用必须具备稳定 call id、请求、结果、状态、时间、关联产物和脱敏后的模型上下文记录，支持取消、幂等、崩溃恢复和任务历史复盘。
5. 旧 `TodoPlan`、`EditingPlan`、固定验证/修复 Graph 只能作为历史 checkpoint 与兼容通道，不能作为新编程任务的最终默认主链。

首发工具类别至少覆盖：仓库/目录枚举、文本搜索、聚焦文件读取、结构化文件编辑、命令与验证、Git 状态/Diff、Todo 更新、审批请求和最终交付。

## 2. 上线 MVP 的不可妥协原则

### 2.1 不允许 Demo 化

以下情况一律视为未完成：

1. 默认任务来自 fixture。
2. 新建任务后跳到硬编码 task id。
3. 任务详情默认展示 demoDiff、demoDelivery 或假报告。
4. UI 显示成功，但后端没有真实状态和产物。
5. 合并、审批、证据包、隐私账本只有静态展示，没有真实数据。

### 2.2 不允许黑箱 Agent

用户必须知道：

1. Agent 当前在做什么。
2. Agent 读了什么。
3. Agent 发给模型什么。
4. Agent 记住了什么。
5. Agent 为什么请求审批。
6. Agent 为什么认为可以交付。

### 2.3 不允许不可追溯交付

每个完成任务必须可追溯：

1. 用户输入。
2. 运行契约。
3. Agent Todo。
4. 命令记录。
5. 验证结果。
6. 自动修复轮次。
7. 审批记录。
8. Diff。
9. 质量门禁。
10. 交付评分。
11. 隐私账本。
12. 合并记录。

### 2.4 不允许上线后用户找不到数据

用户必须能看到：

1. 数据库路径。
2. worktree 路径。
3. 任务产物路径。
4. 日志占用。
5. 截图占用。
6. 临时上下文占用。
7. 可清理内容。
8. 不可清理的永久证据。

## 3. Codex / Cursor 对标基线

这些是竞争级 MVP 的基础能力，全部必须完成呀。

### 3.1 任务与仓库

1. 选择本地 Git 仓库。
2. 检测仓库合法性、当前分支、脏状态。
3. 创建自然语言编程任务。
4. 为任务创建独立 branch 和 worktree。
5. 保存任务、仓库、branch、worktree、状态、创建时间、更新时间。
6. 支持历史任务列表和任务详情恢复。
7. 支持多任务历史并行存在，不互相污染。

### 3.2 Agent 执行

1. Agent 必须维护可见 Todo，但 Todo 不是固定执行顺序或一次性编辑计划的替代品。
2. 模型每轮可从 Runtime 注册的受控工具中选择下一步，并只依据已返回的真实结果继续决策。
3. Agent 只能在任务 worktree 内读取或修改文件；所有路径均由 Runtime 强制校验。
4. Agent 可以按需搜索、读取、编辑、运行命令、检查 Git 状态/Diff、请求审批和提交交付；不能把任意文本当作 shell 或文件操作执行。
5. 验证失败后，模型必须看到真实失败结果并自主决定读取、分析、修复或请求人工介入，不能依赖预制 repair 指令。
6. 工具轮次、重复调用、Token/时间/输出大小必须可配置且有上限；达到上限后进入人工介入，不允许无限循环。
7. 每轮模型请求、工具请求、工具结果、审批、取消和恢复都写入任务时间线与证据链。
8. Agent 能在应用或进程中断后从 checkpoint 恢复：不得重复执行已完成的副作用调用，也不得丢失待处理的工具请求。

### 3.3 终端与日志

1. 支持本地命令执行。
2. 支持 stdout/stderr 实时捕获。
3. 支持命令取消。
4. 支持日志落盘。
5. 支持大日志分页/增量加载。
6. 支持失败摘要提取。
7. 支持命令安全策略和审批。

### 3.4 Diff、审查与合并

1. 支持只读 Diff 查看。
2. 支持按文件查看新增、修改、删除。
3. 支持测试报告展示。
4. 支持交付总结。
5. 支持合并前检查。
6. 支持本地合并。
7. 合并冲突不能静默覆盖。
8. 合并成功或冲突都必须留下记录。

### 3.5 设置与模型

1. 支持 OpenAI-compatible 模型配置。
2. 支持 Base URL、API Key、模型名。
3. API Key 必须安全存储和脱敏展示。
4. 支持模型连接测试。
5. 支持 Agent 输出语言配置。

## 4. CodeMax 必须领先的产品能力

这些能力是本产品与 Codex / Cursor 对打的核心呀，不做完整就不算最终上线 MVP。

### 4.1 Privacy Ledger 隐私证明账本

每个任务必须生成隐私账本，记录：

1. 本地读取了哪些文件。
2. 哪些文件片段进入模型上下文。
3. 哪些记忆被使用。
4. 哪些日志、Diff、工具结果进入上下文。
5. 哪些内容被脱敏。
6. 哪些内容被阻断。
7. 调用了哪个模型供应商。
8. 是否使用私有模型或本地优先模式。

验收：

1. 任务详情能查看隐私摘要。
2. 交付审查页能查看隐私摘要。
3. Proof Pack 中包含隐私章节。
4. 敏感内容不会明文进入普通日志或证据包。

### 4.2 Run Contract 运行契约

每个任务启动前必须生成运行契约，用户可以确认或修改：

1. 任务模式。
2. 模型。
3. 推理强度。
4. 权限等级。
5. 网络策略。
6. 允许路径。
7. 允许命令。
8. 验证命令。
9. Token 预算。
10. 输出语言。
11. 记忆作用域。
12. 超契约处理策略。

验收：

1. 创建任务前可见运行契约。
2. 契约随任务持久化。
3. Agent 突破契约时必须审批。
4. 交付审查页和 Proof Pack 能追溯契约与实际执行差异。

### 4.3 Memory Cockpit 可控记忆舱

必须提供独立的记忆管理体验：

1. 查看长期记忆。
2. 查看记忆来源。
3. 查看记忆作用域。
4. 编辑记忆。
5. 删除记忆。
6. 停用记忆。
7. 查看最近使用时间。
8. 查看本任务用了哪些记忆。

验收：

1. 用户能管理长期记忆。
2. 删除或停用后，后续任务不再使用。
3. 任务详情能说明本次使用了哪些记忆。

### 4.4 Preference Distiller 偏好蒸馏

系统可以从用户行为中提取候选偏好，但不能静默写入长期记忆：

1. 用户反馈。
2. 审批通过/拒绝。
3. 常用验证命令。
4. 常用提交风格。
5. 常见技术选择。
6. 对 Agent 修改的二次修正。

候选偏好必须支持：

1. 接受。
2. 编辑后接受。
3. 忽略。
4. 拒绝。
5. 不再提示同类偏好。

验收：

1. 候选偏好不会自动进入长期记忆。
2. 用户确认后才写入。
3. 被拒绝偏好不会频繁重复提示。

### 4.5 Personal Profile 个人工程画像

必须支持个人画像配置：

1. 默认画像。
2. 安全优先。
3. 严格测试。
4. 低成本。
5. 快速原型。
6. 深度重构。
7. 自定义画像。

画像至少影响：

1. 默认模型。
2. 权限等级。
3. 验证强度。
4. Token 预算。
5. 质量门禁。
6. 输出语言。
7. 记忆使用范围。

验收：

1. 用户可以切换画像。
2. 任务运行契约能继承画像。
3. 任务详情能说明使用了哪个画像。

### 4.6 Proof Pack 审计级证据包

每个完成任务都必须生成证据包：

1. 任务描述。
2. 运行契约。
3. Todo 计划。
4. 命令记录。
5. 日志摘要。
6. 验证结果。
7. 自动修复轮次。
8. Diff。
9. 审批记录。
10. 隐私账本。
11. 质量门禁。
12. 交付评分。
13. 风险说明。
14. 合并记录。

验收：

1. 交付审查页可导出 Proof Pack。
2. 导出的证据包不包含敏感明文。
3. 清理临时数据后，Proof Pack 仍可查看。

### 4.7 Quality Gate 质量门禁

合并前必须经过质量门禁：

1. 验证命令通过。
2. 无未处理审批。
3. 无未解释失败轮次。
4. 敏感信息扫描通过。
5. Diff 非空且可审查。
6. Proof Pack 可生成。
7. 运行契约无未审批突破。

验收：

1. Gate 未通过时默认不能合并。
2. 强制覆盖必须写原因。
3. 覆盖原因进入任务日志和 Proof Pack。

### 4.8 Delivery Score 交付评分

每个任务必须生成交付评分：

1. 验证覆盖度。
2. Diff 规模。
3. 修复轮次。
4. 风险等级。
5. 审批强度。
6. 契约突破情况。
7. 隐私风险。
8. 用户反馈次数。

验收：

1. 交付审查页显示总分。
2. 显示评分项解释。
3. 评分进入 Proof Pack。

### 4.9 Token Budget / Context Budgeter

必须有上下文预算能力：

1. 不默认读取整仓库。
2. 按最近消息、摘要、记忆、文件片段、工具结果、完整文件分层。
3. 记录关键模型调用 token 使用。
4. 记录上下文来源。
5. 失败、低置信度或风险升高时触发质量兜底。
6. 超预算时按契约处理。

验收：

1. 任务启动前有预算预估。
2. 任务详情有预算摘要。
3. Proof Pack 可追溯关键上下文来源。

### 4.10 Skills / Rules / Hooks

必须提供基础工程规则能力：

1. 内置 Bug 修复规则。
2. 内置补测试规则。
3. 内置安全检查规则。
4. 内置发布前检查规则。
5. 命令前 Hook。
6. 写文件前 Hook。
7. 验证后 Hook。
8. 合并前 Hook。

验收：

1. 规则命中写入任务日志。
2. Hook 扩权必须审批。
3. 规则和 Hook 结果进入 Proof Pack。

### 4.11 Model Arena 多模型对战

必须支持复杂任务的多模型方案对比：

1. 用户可为任务开启多模型方案。
2. 多模型分别输出方案。
3. 系统展示不同方案的影响范围、风险、预计成本。
4. 用户选择方案后再执行。

验收：

1. 至少支持同一 OpenAI-compatible 配置下不同模型对比。
2. 方案选择记录进入任务日志和 Proof Pack。

### 4.12 Task Capsule 任务胶囊

任务完成后必须形成可复盘胶囊：

1. 任务摘要。
2. 关键决策。
3. 关键产物索引。
4. 验证命令。
5. 风险和评分。
6. 可重新打开历史任务。

验收：

1. 历史任务可从任务胶囊恢复查看。
2. 临时数据清理后，胶囊仍可打开。

## 5. 四人并行开发总结构

4 位工程师按领域并行开发，不按时间拆分，不互相等待完整实现。  
先冻结共享契约，再各自推进模块，最后按联调顺序收口。

| 负责人 | 领域 | 最终目标 |
| --- | --- | --- |
| A | 任务主链与 Agent 基线 | 让 CodeMax 具备 Codex 式多轮工具调用核心任务能力 |
| B | 隐私、契约、画像、记忆 | 让 CodeMax 在隐私和个人化上形成优势 |
| C | 交付审查、门禁、评分、证据 | 让 CodeMax 的交付可信、可审、可复盘 |
| D | 桌面产品、设置、存储、打包、验收 | 让 CodeMax 可安装、可配置、可上线 |

## 6. A 负责人：任务主链与 Agent 基线

### A-1 负责范围

1. 仓库选择。
2. 任务创建。
3. worktree/branch 创建。
4. 任务列表。
5. 任务详情。
6. Todo 看板。
7. Agent 多轮工具调用协议、状态和恢复。
8. Runtime 工具注册、结果回灌与事件接入。
9. 命令执行与日志联通。
10. 自动验证与基于真实结果的修复展示。
11. 本地合并基础链路。

### A-2 必交付任务

| 编号 | 任务 | 交付物 | 验收 |
| --- | --- | --- | --- |
| A-01 | 去除任务默认 demo/fixture 路径 | 真数据任务入口 | 新建任务、列表、详情均来自真实数据 |
| A-02 | 真实任务创建链路 | task + branch + worktree + agent session | 创建任务后数据库和文件系统都有真实记录 |
| A-03 | 任务列表真实化 | 任务总览页 | 支持状态筛选、仓库筛选、历史恢复 |
| A-04 | 任务详情真实化 | 任务详情页 | Todo、日志、Diff、交付入口均按真实 task id 加载 |
| A-05 | Codex 式多轮工具调用主链 | tool protocol + autonomous loop | 模型能依据真实工具结果动态决定下一步；不依赖固定 TodoPlan -> EditingPlan -> Validation 序列 |
| A-06 | Runtime 工具执行与回灌 | tool request/result + event timeline | 每个调用具备 call id、Schema 校验、幂等、脱敏、产物引用和恢复语义 |
| A-07 | 自动验证与修复展示 | 修复轮次 UI / 数据 | 每轮失败、分析、修复、再验证由真实工具结果驱动且可追溯 |
| A-08 | 合并基础链路联通 | merge preview / merge action | 本地合并成功或冲突均有记录 |
| A-09 | 多任务隔离 | 多任务历史与 worktree 隔离 | 多个任务不会互相污染 |

### A-3 共享接口要求

A 必须向其他负责人稳定提供：

1. `task_id`
2. `repository_id`
3. `worktree_path`
4. `task_branch`
5. `target_branch`
6. `task_status`
7. `agent_stage`
8. `latest_validation_status`
9. `latest_diff_summary`
10. `merge_preview`
11. `pending_tool_request`
12. `latest_tool_result`
13. `agent_round` / `max_agent_rounds`
14. `tool_call_evidence_refs`

### A-4 完成定义

A 完成后，其他人可以基于真实 task id 接入隐私、证据、门禁、记忆、预算和打包 smoke。

## 11. 并行推进方式

不按天排，不按阶段等人。按契约并行推进。

### 11.1 第一步：共享契约冻结

四个人先对齐：

1. task 状态。
2. event 类型。
3. delivery review 聚合结构。
4. privacy ledger 结构。
5. run contract 结构。
6. proof pack 目录结构。

验收：

1. 每个人可以基于这些结构写自己的模块。
2. 不需要等别人 UI 完成。

### 11.2 第二步：四线并行实现

1. A 做真实任务主链。
2. B 做隐私、契约、画像、记忆。
3. C 做交付、证据、门禁、评分、规则。
4. D 做设置、存储、打包、验收。

验收：

1. 每条线都有真实数据入口。
2. 每条线都能被最终 E2E 串起来。

### 11.3 第三步：主链集成

集成顺序：

1. A + B：任务主链接隐私、契约、记忆。
2. A + C：任务主链接交付审查、门禁、评分。
3. C + D：交付审查接打包、存储、证据导出。
4. 全员：安装包内跑完整任务。

### 11.4 第四步：上线验收

必须在安装后的应用中完成最终验收，不能只在开发环境验收哦。

## 12. 最终验收标准

### 12.1 主链验收

1. 选择真实 Git 仓库。
2. 创建真实任务。
3. 创建真实 branch 和 worktree。
4. Agent 建立或更新可见 Todo。
5. 模型先选择搜索/读取等受控工具，并基于真实返回内容决定后续动作。
6. 模型选择结构化编辑或命令工具；Runtime 校验 worktree、命令策略、隐私、预算和审批。
7. Runtime 将真实执行结果、失败摘要、截断标记和证据引用回灌模型。
8. 验证失败后，模型根据失败结果主动搜索、读取、修复并重新验证；验收轨迹必须证明其后续调用受真实结果影响，而不是固定脚本。
9. 高风险动作进入审批并可用同一 call id 恢复，不能以新调用重复副作用。
10. 生成 Diff。
11. 打开交付审查页。
12. Quality Gate 通过。
13. Delivery Score 生成。
14. Proof Pack 导出。
15. 用户确认后本地合并。
16. 应用或 Agent 进程中断后，任务能恢复待处理调用且不会重复执行已完成调用。
17. 轮次、Token、输出大小或重复调用上限触发时，任务进入带原因的人工介入状态，不会虚报成功。

### 12.2 隐私验收

1. 任务启动前能看到隐私预览。
2. 任务完成后能看到隐私账本。
3. 敏感内容被脱敏或阻断。
4. API Key 不进入普通日志。
5. Proof Pack 不包含敏感明文。

### 12.3 画像验收

1. 用户能查看长期记忆。
2. 用户能删除长期记忆。
3. 候选偏好必须确认后写入。
4. 用户能切换个人画像。
5. 任务说明中能看到本次使用了哪些画像/偏好/记忆。

### 12.4 交付验收

1. 统一交付审查页可作为合并前唯一决策页。
2. Gate 失败不能默认合并。
3. Score 有解释。
4. Risk Radar 有风险来源。
5. Proof Pack 可导出并可复查。
6. Task Capsule 可从历史任务打开。

### 12.5 打包验收

1. 生成 Windows 安装包。
2. 干净环境可安装。
3. 首次启动完成自检。
4. 安装后可配置模型。
5. 安装后可完成真实任务闭环。
6. 数据路径和占用可见。
7. 清理策略有效。

## 13. 上线门禁

以下任一项不满足，不允许上线：

1. 默认路径仍依赖 Demo 或 Fixture。
2. 无法创建真实任务。
3. Agent 运行过程不可见。
4. 高风险动作不会审批。
5. 隐私账本不可用。
6. 运行契约不可用。
7. 个人记忆不可控。
8. Proof Pack 不可导出。
9. Quality Gate 不可用。
10. Delivery Score 不可用。
11. 交付审查页不能做最终合并决策。
12. 安装包无法完成真实任务 smoke。
13. 敏感信息进入普通日志或导出文件。
14. 存储路径、占用、清理策略对用户不可见。
15. 编程模式默认仍是固定 `TodoPlan -> EditingPlan -> Validation` 或预制 repair 序列，而不是模型根据真实工具结果持续决策的多轮工具调用主链。
16. 工具调用无法校验 Schema、路径、命令、审批、预算、隐私或幂等，或工具结果不能回灌模型并形成证据。
17. 应用或 Agent 重启后会重复执行副作用调用、丢失待处理调用，或无法恢复任务历史。

## 15. 最终结论

本次开发不是做 Demo，不是补一个小版本，也不是只完成“V1 必做项”。  
本次目标是直接完成一个可上线、可安装、可对打 Codex / Cursor 的竞争级 MVP 应用。

四位工程师按以下方向并行：

1. A：把 Codex 式多轮工具调用任务主链做真。
2. B：把隐私保护和个人画像做强。
3. C：把交付审查和证据门禁做可信。
4. D：把桌面产品和上线交付做完整。

最终 CodeMax 必须做到：

`模型能根据真实工具结果自主推进任务，而不是照固定节点序列执行；每次读取、编辑、命令、审批和交付都受 Runtime 控制并可审计；会记住用户，也允许用户控制；能给出可信交付证据；能在开发环境跑，也能安装给真实用户使用。`
