# Agent 编程任务调度台最终需求文档

## 1. 项目概述

### 1.1 产品名称

Agent 编程任务调度台。

### 1.2 产品定位

本产品是一款面向 AI 编程 Agent 的桌面端任务管理与执行调度工具。它不是传统 IDE，也不是单纯的聊天机器人，而是一个用于管理、观察、审批、验证和合入 AI 编程任务的本地指挥中心。

用户通过自然语言创建编程任务，系统在隔离工作区中调度 Agent 自动完成任务理解、任务拆解、代码修改、命令执行、编译测试、自我修复和交付总结。最终，用户通过桌面端审查 Diff、测试结果、运行日志和 Agent 说明，再决定是否合入主代码库。

### 1.3 一句话说明

把 AI 当作远程编程员工，把桌面端当作任务指挥部、本地沙箱、审批中心和交付验证台。

## 2. 建设目标

### 2.1 核心目标

1. 支持用户用自然语言创建编程任务。
2. 支持一个或多个 Agent 任务并行运行。
3. 每个任务必须运行在独立隔离工作区中。
4. Agent 必须先拆解任务，再执行代码修改。
5. 系统必须能执行本地编译、测试、脚本命令并捕获日志。
6. 系统必须支持“执行失败 -> 分析错误 -> 自动修复 -> 再次验证”的闭环。
7. 高风险操作必须进入人工审批流程。
8. 交付时必须提供 Diff、测试报告、运行日志和 Agent 总结。
9. 用户确认后才允许合入主代码库。
10. 系统必须通过分层存储、滚动摘要和清理策略减少本地占用，同时不影响任务继续、历史审查和合入。

### 2.2 MVP 目标

MVP 阶段实现单仓库、单用户、本地运行的完整闭环：

```text
选择本地 Git 仓库
  -> 创建自然语言任务
  -> 创建 Git Worktree 隔离工作区
  -> Agent 分析并生成任务清单
  -> Agent 修改代码
  -> 后台执行验证命令
  -> 失败后自动修复，最多 5 轮
  -> 成功后展示 Diff 和测试结果
  -> 用户确认后合入主代码库
```

### 2.3 产品边界

MVP 阶段不做完整 IDE，不提供云端协作，不自动推送远程仓库，不绕过用户审批执行危险命令，也不默认自动合入主分支。

### 2.4 差异化亮点定位

市面上的 AI 编程工具已经普遍具备代码生成、Agent 执行、Diff 审查、云端任务、规则、记忆、子代理、Hooks 或 Worktree 隔离等能力。本产品不能只做“另一个 AI 编程工具”，也不应把竞争重点放在“谁更会写代码”上，而应把差异化建立在“更懂用户、更安全、更可控、更可验证、更节省、更适合本地长期使用”上。

产品核心表达应调整为：

```text
不是更会写代码的 Agent，而是更懂你、更守规矩、更可验证的本地 AI 工程调度台。
```

核心亮点如下：

| 亮点 | 产品表达 | 用户价值 |
| --- | --- | --- |
| 审计级证据包 Proof Pack | 每个任务交付完整证据：需求、计划、命令、日志、测试、截图、Diff、审批、风险说明 | 用户不只看到结果，还能知道 Agent 为什么这样改、怎么验证过 |
| 质量门禁 Quality Gate | 合入前必须经过测试、构建、风险、密钥、审批等门禁检查 | 降低 AI 代码“看起来能用但实际有坑”的风险 |
| 隐私证明账本 Privacy Ledger | 任务运行前后记录哪些文件、上下文、记忆和密钥被读取、脱敏、发送或阻断 | 让用户知道数据去了哪里，避免 AI 工具成为隐私黑箱 |
| 可控记忆 Memory Cockpit | 用户可查看、编辑、删除长期记忆；记忆来源和作用域透明 | 解决 AI 记忆黑箱问题，避免错误偏好长期污染任务 |
| 个人喜好蒸馏 Preference Distiller | 从用户反馈、审批、拒绝、常用命令和修改习惯中提炼候选偏好，必须经用户确认后才进入长期记忆 | 让 Agent 越用越懂用户，但不擅自替用户做决定 |
| 个人工程画像 Personal Profile | 用户可建立“保守、安全优先、低成本、严格测试、快速原型”等画像，并绑定模型、权限、验证和语言偏好 | 同一个工具能适应不同用户和不同项目习惯 |
| 运行契约 Run Contract | 在任务对话框中统一切换模式、模型、思考强度、权限、联网、预算、验证命令和输出语言 | 用户在任务开始前就知道 Agent 可以做什么、不可以做什么 |
| 不降智省 Token Context Budgeter | 通过 AST 检索、上下文分层、滚动摘要、记忆筛选和按需升档减少 Token，同时保留质量兜底 | 降低成本和等待时间，但不牺牲复杂任务质量 |
| 本地低占用 Local Slim Mode | SQLite 只存索引和摘要，大文件落盘并可清理，仓库用 Worktree 不复制 | 长期使用不会越来越臃肿 |
| 内置 Skills 与规则包 Skill/Rule Packs | 内置常见工程流程、代码审查、安全检查、测试补全、UI 验证、发布检查等可启用规则包 | 用户不用每次重复写提示词，团队规范也可沉淀 |
| Hooks Studio | 支持在任务启动、命令前、写文件前、验证后、合入前等生命周期配置钩子 | 让用户和团队能用机械规则兜住 Agent 行为 |
| 任务胶囊 Replayable Task Capsule | 任务完成后保留可复盘的摘要、证据和关键产物，可导出给团队或自己复查 | 适合技术负责人、代码审查和事故追踪 |
| 多模型对战 Model Arena | 同一复杂任务可让多个模型分别产出方案或补丁，由用户或评审 Agent 选择 | 不绑定单一模型，提升复杂任务成功率 |
| 风险雷达 Risk Radar | 执行前识别危险命令、敏感文件、越权路径、依赖变更、Schema 变更 | 相比默认放权的 Agent 更稳、更安全 |
| 交付评分 Delivery Score | 根据验证结果、覆盖范围、风险、Diff 规模、审批情况给出交付评分 | 用户快速判断是否值得合入 |
| 超多国际化 i18n-first | UI 文案、Agent 输出、任务模板、规则说明、证据包和错误提示均支持多语言 | 面向长期分发和多语言用户，而不是只服务单一语言场景 |

完整版必须把“隐私可证明、记忆可控、运行可配置、交付可审计、成本可解释、规则可扩展、语言可切换”作为产品主轴。MVP 阶段可以先跑通核心闭环，但最终结果必须具备上述亮点，否则容易沦为普通 AI 编程调度工具。

## 3. 目标用户

| 用户类型 | 使用目的 | 核心诉求 |
| --- | --- | --- |
| 个人开发者 | 修复 Bug、补充测试、局部重构 | 快速完成任务，同时保护当前工作区 |
| 技术负责人 | 同时管理多个 AI 编程任务 | 掌握进度、质量、风险和交付结果 |
| 架构师 | 审查复杂修改方案 | 需要多方案对比、影响范围和审批能力 |
| 测试 / QA | 查看验证结果 | 需要测试日志、失败原因和运行截图 |

## 4. 核心使用场景

### 4.1 修复本地 Bug

用户选择代码仓库，输入“修复支付模块金额精度 Bug”。系统创建隔离工作区，Agent 分析相关文件并生成 Todo List，修改代码后运行测试。测试失败时，Agent 根据错误日志继续修复。测试通过后，系统展示 Diff 和验证报告，用户确认后合入。

### 4.2 补充单元测试

用户输入“为商品服务补充核心单元测试”。Agent 分析项目测试框架、服务层代码和已有测试风格，生成测试用例并运行验证命令。用户审查新增测试文件和执行结果后决定是否合入。

### 4.3 多任务并行

用户同时创建多个任务，例如修复 Bug、补测试、优化页面。每个任务使用独立 Worktree、独立日志、独立状态机和独立交付物，互不污染。

### 4.4 高风险操作审批

Agent 准备执行数据库迁移、删除文件、安装新依赖、执行危险命令或合入主分支时，任务自动挂起。用户在审批弹窗中查看命令、执行目录、风险说明和 Agent 理由，然后选择允许、拒绝或要求修改。

### 4.5 复杂重构方案选择

遇到架构重构时，Agent 先输出 2-3 个方案，说明改动范围、优缺点、风险和推荐理由。用户选择方案后，Agent 才继续执行。

### 4.6 隐私可见的任务启动

用户输入任务后，系统在真正调用模型前展示本次任务预计读取的文件范围、会发送给模型的上下文摘要、会使用的长期记忆、是否包含敏感路径或密钥痕迹、是否需要脱敏以及将调用的模型供应商。用户可以确认、缩小上下文、切换本地或私有模型、关闭某些记忆，或要求 Agent 重新规划。

### 4.7 个人模式与运行契约切换

用户可以在任务对话框中切换“快速原型、严格测试、安全优先、低成本、深度思考、自定义”等运行模式。每个模式会影响模型、思考强度、权限、联网、自动修复轮次、验证命令、Token 预算和交付要求。系统必须在任务开始前把这些选择汇总为一份运行契约。

### 4.8 不降智省 Token

面对大型仓库时，系统不能默认把整仓塞进上下文。Agent 应先通过文件索引、Git Diff、Tree-sitter、最近消息、滚动摘要和相关记忆筛选上下文；当置信度不足、验证失败或风险升高时，再按规则追加上下文、升级模型或请求用户确认。

### 4.9 内置 Skills、规则与钩子

用户可以启用内置工程能力，例如 Bug 修复流程、测试补全流程、安全审查流程、前端视觉验收流程、发布前检查流程。用户也可以配置 Hooks，在命令执行前、文件写入前、依赖变更前、合入前自动拦截或追加检查。

### 4.10 多语言与个人化交付

用户可以选择界面语言、Agent 回复语言、证据包语言、错误提示语言和提交信息语言。系统默认简约 UI，但必须支持多套 UI 风格和语言包，且 UI 设计方案需要先征询用户意见后再落地。

## 5. 功能需求

### 5.1 仓库管理

1. 用户可以选择本地 Git 仓库。
2. 系统校验仓库是否合法。
3. 系统展示仓库名称、路径、当前分支、未提交改动状态。
4. 系统记录最近使用仓库。
5. MVP 阶段仅支持本地 Git 仓库。

### 5.2 任务创建

1. 用户可以输入自然语言任务描述。
2. 用户可以选择任务类型：修复 Bug、补充测试、局部重构、解释代码、自定义任务。
3. 系统为任务生成唯一 ID。
4. 系统为任务创建独立工作区和独立分支。
5. 创建完成后进入任务详情页。

### 5.3 任务列表

1. 展示全部任务。
2. 展示任务名称、状态、所属仓库、创建时间、最近更新时间。
3. 支持按状态筛选。
4. 支持查看进行中、等待审批、需要人工介入、已完成、已合入、失败、已取消任务。
5. 支持进入任务详情。

### 5.4 任务拆解看板

1. Agent 接收任务后必须先生成 Todo List。
2. 每个 Todo 包含标题、说明、状态、开始时间、完成时间。
3. Todo 状态包括待执行、执行中、已完成、失败、等待审批。
4. Agent 每完成一个阶段，前端实时更新状态。
5. 用户可以查看每个 Todo 对应的日志和说明。

### 5.5 隔离工作区

1. 系统使用 `git worktree` 创建隔离工作区。
2. 每个任务拥有独立目录和独立分支。
3. Agent 只能在任务工作区中修改代码。
4. Agent 的中间修改不得污染用户主工作区。
5. 工作区路径、分支名和任务 ID 必须持久化保存。
6. 用户可以在任务完成或取消后手动清理工作区。

### 5.6 命令执行

1. 系统支持在隔离工作区执行本地命令。
2. 命令包括编译、测试、安装依赖、启动项目、代码检查等。
3. 系统实时捕获 Stdout、Stderr、退出码和耗时。
4. 命令执行日志按时间顺序展示。
5. 用户可以取消长时间运行命令。

### 5.7 自动验证与自我修复

1. Agent 修改代码后必须运行验证命令。
2. 验证命令由系统自动识别并允许用户确认或修改。
3. Node.js 项目默认识别 `package.json` 中的 test、lint、build 脚本。
4. Python 项目默认识别 pytest、unittest、ruff、mypy 等常见命令。
5. Java 项目默认识别 Maven 或 Gradle 命令。
6. 验证失败后，系统将错误日志和相关代码上下文交给 Agent 重新分析。
7. Agent 自动修复后再次执行验证。
8. 默认最多自动修复 5 轮。
9. 超过最大轮次仍失败时，任务进入“需要人工介入”状态。

### 5.8 人工审批

1. 高风险操作前任务必须挂起。
2. 审批弹窗展示操作内容、执行目录、风险等级、影响范围和 Agent 理由。
3. 用户可以选择允许、拒绝或要求修改。
4. 用户审批结果必须写入任务日志。
5. 未审批前，高风险操作不得执行。

高风险操作包括：

1. 删除文件或目录。
2. 修改数据库 Schema。
3. 安装、升级或删除依赖。
4. 执行包含 `rm`、`del`、`format`、`sudo`、`chmod -R` 等危险命令。
5. 推送远程仓库。
6. 合入主代码库。
7. 修改系统目录或用户主目录外的非任务工作区文件。

### 5.9 方案选择

1. 复杂重构任务必须先给出方案。
2. 方案数量为 2-3 个。
3. 每个方案必须包含目标、改动范围、优点、缺点、风险和推荐理由。
4. 用户选择方案后，Agent 按所选方案执行。
5. 用户可以要求 Agent 重新生成方案。

### 5.10 Diff 审查

1. 系统提供只读 Diff 视图。
2. 支持按文件查看新增、修改、删除内容。
3. 支持代码高亮。
4. 支持折叠未修改代码。
5. 支持查看 Agent 对单个文件改动的解释。
6. 用户可以基于 Diff 输入反馈，让 Agent 继续修改。

### 5.11 测试报告

1. 展示所有执行过的验证命令。
2. 展示每条命令的执行状态、退出码、耗时。
3. 展示测试通过或失败摘要。
4. 展示关键错误日志。
5. 前端项目支持保留运行截图。

### 5.12 交付说明

Agent 完成任务后必须生成交付说明，内容包括：

1. 本次任务解决的问题。
2. 关键修改点。
3. 变更文件列表。
4. 已执行的验证命令。
5. 验证结果。
6. 剩余风险。
7. 建议 Commit Message。

### 5.13 一键合入

1. 用户审查通过后可以点击合入。
2. 合入前必须展示最终 Diff、测试结果和提交信息。
3. 系统默认将 Agent 分支合入当前目标分支。
4. 合入方式采用本地 Git 合并，不自动推送远程仓库。
5. 合入成功后生成本地 Commit。
6. 合入失败时展示冲突文件和错误原因。
7. 冲突未解决前不得标记为合入成功。

### 5.14 存储瘦身与对话记忆

1. 系统采用“SQLite 存结构化索引，文件系统存大文件产物”的分层存储策略。
2. SQLite 只保存任务、状态、审批、摘要、路径、配置和轻量消息，不直接保存大日志、大 Diff、大截图和完整仓库副本。
3. 原始日志、Diff Patch、测试报告、截图、运行产物统一保存在任务产物目录中，并在 SQLite 中保存路径引用。
4. 任务代码隔离统一使用 Git Worktree，不使用全量复制仓库作为默认方案。
5. 对话记忆采用“最近消息原文 + 历史滚动摘要 + 结构化长期记忆”的方式。
6. 每个任务默认保留最近 50 条可见对话原文，超过窗口的历史消息自动压缩为摘要。
7. 长期记忆只保存用户偏好、仓库配置、任务关键决策、审批结果、默认验证命令等可复用信息。
8. Agent 中间草稿、临时上下文和大段原始模型交互默认短期保存，任务完成后摘要化或清理。
9. 用户可以在设置页配置日志、截图、Worktree、临时上下文和历史消息的保留周期。
10. 用户可以查看、删除或清空长期记忆和指定任务的对话记忆。
11. 系统不得保存模型不可见的内部推理过程；只保存用户可见消息、操作记录、摘要、决策和结果。
12. 清理历史产物时必须保留任务摘要、最终 Diff、测试报告摘要、审批记录和合入记录，确保历史审查不受影响。

### 5.15 差异化亮点功能

#### 5.15.1 审计级证据包 Proof Pack

1. 每个任务完成后必须生成一个证据包。
2. 证据包包含任务原始描述、Agent 计划、Todo List、关键命令、命令结果、关键日志、测试报告、最终 Diff、审批记录、风险说明和交付总结。
3. 证据包支持在任务详情页查看。
4. 证据包支持导出为本地目录或压缩包。
5. 清理临时产物后，证据包仍必须保留最终审查所需的摘要和关键路径。

#### 5.15.2 质量门禁 Quality Gate

1. 合入前必须执行质量门禁检查。
2. 默认门禁包括验证命令通过、无未审批高风险操作、无明显密钥泄露、最终 Diff 可生成、交付说明完整。
3. 用户可以配置门禁项，例如必须通过 test、lint、build、覆盖率阈值、禁止修改文件范围等。
4. 门禁未通过时，系统不得默认合入。
5. 用户强制覆盖门禁时，必须二次确认并记录原因。

#### 5.15.3 可控记忆 Memory Cockpit

1. 用户可以查看系统记住了哪些用户偏好、仓库配置、任务决策和审批结果。
2. 每条长期记忆必须展示来源、作用域、更新时间和是否可编辑。
3. 用户可以编辑或删除长期记忆。
4. Agent 使用长期记忆时，应在任务上下文摘要中说明使用了哪些记忆。
5. 被用户删除的记忆不得再次被自动加载。

#### 5.15.4 任务胶囊 Replayable Task Capsule

1. 任务完成后生成可复盘任务胶囊。
2. 任务胶囊包含任务摘要、关键决策、环境信息、验证命令、最终 Diff、证据包索引和交付评分。
3. 用户可以在历史任务中重新打开任务胶囊。
4. 增强版支持基于任务胶囊重新运行验证命令或重新生成交付说明。

#### 5.15.5 多模型对战 Model Arena

1. 对复杂重构或高风险任务，用户可以选择让多个模型生成候选方案。
2. 系统展示不同模型的方案、风险、影响范围和预估成本。
3. 用户可以选择一个方案执行，也可以要求评审 Agent 给出推荐。
4. 增强版支持让多个模型分别生成补丁，并基于验证结果进行评分。

#### 5.15.6 交付评分 Delivery Score

1. 系统为每个任务生成交付评分。
2. 评分参考验证结果、失败修复次数、Diff 规模、风险操作数量、审批情况、测试覆盖和交付说明完整度。
3. 评分不得替代用户审查，只用于辅助判断。
4. 低评分任务应提示用户谨慎合入。

#### 5.15.7 隐私证明账本 Privacy Ledger

1. 每个任务必须生成隐私账本，记录任务读取的文件范围、使用的记忆、发送给模型的上下文类型、模型供应商、脱敏状态和阻断记录。
2. 隐私账本必须区分“本地读取”“发送到模型”“仅保存摘要”“已脱敏”“已阻断”五类数据流向。
3. 任务启动前，用户必须能预览高风险数据项，例如 `.env`、密钥、证书、私有配置、用户主目录路径和大型日志。
4. 用户可以在任务级别关闭某些记忆、缩小上下文范围、切换模型供应商或启用更严格的隐私模式。
5. 隐私账本必须进入 Proof Pack，便于任务完成后追溯。

#### 5.15.8 个人喜好蒸馏 Preference Distiller

1. 系统可以从用户的自然语言反馈、审批选择、拒绝原因、常用验证命令、提交信息风格和清理偏好中生成候选偏好。
2. 候选偏好不得直接进入长期记忆，必须先展示给用户确认。
3. 每条候选偏好必须展示来源、置信度、影响范围、建议保存位置和示例效果。
4. 用户可以接受、编辑、忽略或永久拒绝候选偏好。
5. 被拒绝的候选偏好不得频繁重复提示，除非出现新的强证据。

#### 5.15.9 个人工程画像 Personal Operating Profile

1. 系统支持多个个人画像，例如默认、保守、安全优先、低成本、严格测试、快速原型、深度重构、自定义。
2. 每个画像可以绑定默认模型、思考强度、Token 预算、权限级别、联网策略、验证命令、质量门禁、输出语言和交付格式。
3. 画像可以按全局、仓库、任务类型三个作用域生效。
4. 用户可以复制、编辑、导入、导出画像。
5. Agent 执行任务时必须记录本次使用的画像和被覆盖的配置项。

#### 5.15.10 运行契约 Run Contract

1. 每个任务创建时必须生成运行契约。
2. 运行契约包括任务模式、模型、思考强度、权限、联网、可写路径、可执行命令、Token 预算、验证命令、记忆加载范围、语言和交付要求。
3. 用户可以在任务启动前修改运行契约。
4. Agent 执行过程中如需突破运行契约，必须发起审批或要求用户修改契约。
5. 运行契约必须持久化，并写入 Proof Pack。

#### 5.15.11 不降智省 Token Context Budgeter

1. 系统必须按“最近对话、滚动摘要、相关记忆、相关文件片段、工具结果、完整文件”分层组织上下文。
2. Agent 默认只加载与任务相关的最小上下文，禁止默认读取整个仓库。
3. 当上下文不足、验证失败、风险升高或用户要求深度分析时，系统可以按规则追加上下文或升级模型。
4. 系统必须记录每次模型调用的输入 Token、输出 Token、上下文来源和节省策略。
5. 省 Token 策略不得跳过验证、审批、质量门禁和交付证据。

#### 5.15.12 内置 Skills 与规则包

1. 系统内置常见任务 Skills，例如 Bug 修复、补测试、重构方案、代码解释、安全审查、前端截图验证、文档生成、发布检查。
2. 每个 Skill 必须定义适用场景、输入要求、执行步骤、默认验证命令、产物要求和风险点。
3. 规则包可以约束代码风格、目录边界、提交信息、测试要求、禁止文件范围、敏感词和审批策略。
4. 用户可以按全局、仓库和任务类型启用或禁用 Skills 与规则包。
5. 规则包命中情况必须进入任务日志和 Proof Pack。

#### 5.15.13 Hooks Studio

1. 系统支持任务启动前、模型调用前、命令执行前、文件写入前、依赖变更前、验证后、生成交付物前、合入前等生命周期钩子。
2. Hook 可以执行检查、阻断、追加命令、追加上下文、要求审批或生成提示。
3. Hook 必须支持启用、禁用、排序、作用域、失败策略和执行日志。
4. 高风险 Hook 修改必须二次确认。
5. Hook 执行结果必须进入任务日志、质量门禁和 Proof Pack。

#### 5.15.14 个性化与个人习惯

1. 系统必须支持对用户常用仓库、常用任务类型、常用验证命令、语言偏好、提交风格、审查习惯和清理习惯进行结构化管理。
2. 个性化必须可解释、可编辑、可删除，不得以黑箱方式影响 Agent 决策。
3. 用户可以临时关闭个性化，以便执行一次性客观任务。
4. 个性化应用到任务时，系统必须在任务上下文摘要中说明使用了哪些习惯。
5. 个性化策略必须尊重运行契约和隐私设置。

#### 5.15.15 国际化与多 UI 风格

1. UI 文案、菜单、错误提示、任务模板、规则说明、证据包、交付说明和 Agent 输出必须支持国际化。
2. 默认 UI 风格为简约、克制、信息密度高，适合长期工程使用。
3. 系统预留多套 UI 风格能力，例如简约、专业、暗色、高对比、紧凑、演示模式。
4. UI 风格落地前必须先征询用户意见，不得在未确认设计方向时直接定稿。
5. 国际化资源必须支持缺失检查、回退语言和导出。

## 7. 状态流转

### 7.1 任务状态

| 状态 | 说明 |
| --- | --- |
| 契约确认中 | 任务已创建草稿，等待用户确认运行契约、隐私范围和预算 |
| 已创建 | 任务已创建，尚未开始分析 |
| 分析中 | Agent 正在理解任务和检索代码上下文 |
| 规划中 | Agent 正在生成 Todo List 或执行方案 |
| 执行中 | Agent 正在修改代码或准备命令 |
| 验证中 | 系统正在执行编译、测试或检查命令 |
| 自动修复中 | 验证失败后，Agent 正在修复问题 |
| 等待审批 | 任务因高风险操作或方案选择而挂起 |
| 需要人工介入 | 自动修复失败或出现不可自动处理的问题 |
| 已完成 | Agent 已交付结果，等待用户审查 |
| 合入中 | 系统正在合入改动 |
| 已合入 | 改动已成功合入主代码库 |
| 已取消 | 用户取消任务 |
| 失败 | 任务异常终止 |

### 7.2 主流程

```text
选择仓库
  -> 创建任务
  -> 确认运行契约
  -> 确认隐私账本预览
  -> 创建 Worktree
  -> Agent 分析任务
  -> 生成 Todo List
  -> 修改代码
  -> 执行验证命令
  -> 验证失败：自动修复
  -> 验证成功：生成交付物
  -> 生成证据包、隐私账本和交付评分
  -> 用户审查
  -> 用户确认合入
  -> 本地 Git 合并
```

### 7.3 审批流程

```text
Agent 申请高风险操作
  -> 任务挂起
  -> 创建审批请求
  -> 用户查看风险和影响范围
  -> 用户允许：继续执行
  -> 用户拒绝：终止该操作并重新规划
  -> 用户要求修改：Agent 重新生成方案
```

## 9. 技术栈说明

### 9.1 总体技术架构

本产品采用桌面端壳体、前端 UI、Rust 本地后台、Python Agent 引擎分层架构。

```text
Tauri Desktop App
  ├─ Frontend UI: React + TypeScript + Tailwind CSS + shadcn/ui
  ├─ Diff Viewer: Monaco Editor
  ├─ Local Backend: Rust + Tauri Commands + Tokio
  ├─ Local Storage: SQLite
  ├─ Artifact Storage: Local File System + Compressed Logs
  ├─ Conversation Memory: Recent Messages + Rolling Summary + Memory Items
  ├─ Privacy Ledger: Data-flow Records + Redaction Logs
  ├─ Run Contract: Mode + Model + Permission + Budget Policy
  ├─ Context Budgeter: AST Retrieval + Summary + Token Accounting
  ├─ Skills/Rules/Hooks: Built-in Packs + Lifecycle Guards
  ├─ I18n: Locale Resources + Fallback Checks
  ├─ Agent Engine: Python + LangGraph
  ├─ Code Parser: Tree-sitter
  ├─ Git Isolation: Git Worktree
  └─ Model Providers: OpenAI-compatible API / Claude / DeepSeek
```

### 9.2 桌面端

| 模块 | 技术选型 | 用途 |
| --- | --- | --- |
| 桌面壳体 | Tauri v2 | 打包桌面应用，提供系统能力和 IPC |
| 本地后台 | Rust | 管理文件、进程、Git、命令执行和安全策略 |
| 异步运行时 | Tokio | 执行异步命令、日志流和后台任务 |
| 进程执行 | tokio::process / portable-pty | 执行编译、测试、脚本命令并捕获输出 |
| 本地存储 | SQLite | 保存任务、摘要、审批、记忆索引、交付物路径和 Agent 状态 |
| 本地产物存储 | 文件系统 + gzip/zstd 压缩 | 保存大日志、Diff Patch、截图、报告和临时上下文 |

选择 Tauri + Rust 的原因：桌面端体积小、性能高、系统能力强，适合处理本地文件、Git Worktree、进程执行和安全审批这类重 I/O 场景。

### 9.3 前端 UI

| 模块 | 技术选型 | 用途 |
| --- | --- | --- |
| 前端框架 | React 18 | 构建任务看板、详情页、审批流 |
| 开发语言 | TypeScript | 提升复杂状态和数据结构的可靠性 |
| 构建工具 | Vite | 提供快速开发和构建能力 |
| 样式方案 | Tailwind CSS | 快速构建一致的界面样式 |
| 组件库 | shadcn/ui | 提供表格、弹窗、按钮、Tabs、表单等组件 |
| 状态管理 | Zustand | 管理前端任务状态、筛选和界面状态 |
| 数据请求 | TanStack Query | 管理异步请求、缓存和刷新 |
| Diff 展示 | Monaco Editor Diff Mode | 展示只读代码 Diff 和高亮 |

选择 React + TypeScript + Tailwind + shadcn/ui 的原因：适合构建任务总览、审批中心、日志面板、Diff 审查等复杂桌面仪表盘界面，开发效率和可维护性都较好。

### 9.4 Agent 引擎

| 模块 | 技术选型 | 用途 |
| --- | --- | --- |
| Agent 语言 | Python 3.11+ | 快速集成模型、工具链和状态机 |
| 状态机 | LangGraph | 管理规划、修改、验证、修复、审批等有状态流程 |
| 服务接口 | FastAPI / 本地 HTTP | 与 Rust 后台进行本地通信 |
| 代码解析 | Tree-sitter | 提取类、函数、依赖、导入和局部上下文 |
| 模型适配 | OpenAI-compatible SDK | 兼容 OpenAI、Claude、DeepSeek 等模型供应商 |
| 持久化 | SQLite Checkpoint | 保存 Agent 状态，支持中断和恢复 |
| 对话记忆 | 滚动摘要 + 结构化记忆 | 保存最近消息、历史摘要、用户偏好、仓库记忆和任务决策 |

选择 Python + LangGraph 的原因：Agent 工作流不是线性链路，而是需要循环、条件分支、暂停审批和状态恢复。LangGraph 可以自然表达“规划 -> 修改 -> 验证 -> 报错 -> 修复 -> 再验证”的闭环。

### 9.5 Git 与隔离方案

| 能力 | 技术选型 | 用途 |
| --- | --- | --- |
| 工作区隔离 | Git Worktree | 为每个任务创建独立目录和分支 |
| 变更查看 | git diff | 生成任务维度 Diff |
| 合入 | git merge / git apply | 用户确认后合入目标分支 |
| 冲突检测 | Git 原生命令 | 检测并展示冲突文件 |

选择 Git Worktree 的原因：相比复制整个项目，Worktree 更快、更节省空间，也不会复制 `node_modules`、`target` 等大型目录，适合大型代码仓库。

### 9.6 模型选择

MVP 阶段支持 OpenAI-compatible API 协议，并提供以下模型供应商配置入口：

1. OpenAI 模型，用于通用推理和代码修改。
2. Claude 模型，用于长上下文推理和复杂重构。
3. DeepSeek Coder，用于高性价比代码任务和国内部署场景。
4. 私有化 OpenAI-compatible 服务，用于企业内网部署。

系统不强绑定单一模型，模型配置包括 Base URL、API Key、模型名、上下文长度和任务默认模型。

### 9.7 截图与前端验证

| 能力 | 技术选型 | 用途 |
| --- | --- | --- |
| 浏览器自动化 | Playwright | 前端项目运行后截图和基础交互验证 |
| 截图存档 | 本地文件系统 | 保存任务交付截图 |
| 报告展示 | 前端报告面板 | 展示截图、日志和执行结果 |

MVP 可先支持基础截图，后续扩展为完整 UI 自动化验证。

### 9.8 存储与对话记忆方案

系统采用本地优先、低占用、可清理的存储架构：

```text
app-data/
  app.db
  tasks/
    task-001/
      logs/
      artifacts/
      diff.patch
      report.json
      screenshots/
      context/
  worktrees/
    task-001/
```

存储规则：

1. SQLite 保存结构化数据、轻量消息、摘要、路径引用和状态，不保存大体积二进制或长日志。
2. 原始日志、截图、Diff Patch、报告和临时上下文保存到文件系统。
3. 日志超过阈值后自动压缩，优先使用 gzip 或 zstd。
4. 已合入任务默认提示清理 Worktree，但保留最终 Diff、测试报告摘要、审批记录和合入记录。
5. 已取消或失败任务默认保留 7 天临时上下文，之后只保留摘要和关键错误。
6. 最近对话原文默认保留 50 条，历史消息由 Agent 生成滚动摘要。
7. 长期记忆只保存用户明确偏好、仓库默认命令、方案选择、审批决策等对后续任务有价值的信息。
8. 用户可以在设置页查看和删除长期记忆。
9. 如后续需要语义检索，可在增强版接入 `sqlite-vec` 或 `sqlite-vss`，MVP 阶段不强制使用向量库。

## 10. 安全要求

1. Agent 默认只能读写任务 Worktree。
2. 用户主工作区默认只读，合入阶段除外。
3. 高风险命令必须审批。
4. 黑名单命令必须禁止执行。
5. API Key 必须加密存储或使用系统凭据能力保存。
6. API Key 不得出现在普通日志、错误提示和交付报告中。
7. 合入前必须展示最终 Diff。
8. 合入前必须二次确认。
9. 合入失败不得静默覆盖用户代码。
10. 对话记忆必须可被用户查看和删除。
11. 系统不得长期保存不可见内部推理过程。
12. 清理策略不得删除最终 Diff、审批记录和合入记录。
13. 任务启动前必须展示隐私账本预览，明确哪些数据会发送给模型。
14. 涉及密钥、证书、环境变量、用户主目录和私有配置的上下文必须默认脱敏或阻断。
15. 长期记忆、候选偏好和个人画像不得在用户不知情的情况下跨仓库滥用。
16. Agent 试图突破运行契约时必须触发审批。
17. Hook 和规则包不得静默扩大 Agent 权限。
18. 导出的证据包、任务胶囊和日志必须支持敏感信息扫描与脱敏。

## 11. 性能要求

1. 任务列表打开时间不超过 1 秒。
2. 任务详情页首次渲染不超过 2 秒。
3. 日志必须增量加载，避免大日志卡顿。
4. Diff 大文件必须支持折叠或按需加载。
5. 创建 Worktree 的时间应明显短于全量复制仓库。
6. Agent 上下文检索不得默认读取整个仓库。
7. SQLite 单库不应被原始日志、截图和大 Diff 撑大。
8. 对话恢复时只加载最近消息、摘要和相关记忆，避免一次性加载全部历史。
9. Token 预算统计必须异步写入，不得阻塞任务执行。
10. 上下文筛选必须支持按需加载，避免一次性解析大型仓库。
11. 长期记忆和候选偏好列表必须分页加载。
12. 国际化资源加载必须支持命名空间拆分，避免首屏加载全部语言包。
13. Skills、规则包和 Hooks 的命中检查必须可缓存，避免每次命令执行都全量扫描。

## 12. 稳定性要求

1. 单个任务失败不得影响其他任务。
2. 应用重启后可以恢复任务列表、日志、审批记录和交付物。
3. Agent 状态必须可持久化。
4. 命令执行中断后必须记录退出状态。
5. 自动修复循环必须有次数上限。
6. 任务取消后不得遗留正在运行的后台进程。
7. 清理 Worktree、日志和截图前必须保证任务摘要、最终产物索引和关键记录已持久化。
8. 应用重启后可以恢复最近对话、任务摘要和长期记忆。
9. 应用重启后可以恢复运行契约、隐私账本、候选偏好和 Token 预算记录。
10. Hook 执行失败时必须按失败策略处理，并记录可追踪日志。
11. 国际化资源缺失时必须回退到默认语言，不得导致页面崩溃。
12. 个人画像损坏或配置缺失时必须回退到默认画像。

11. 任务启动前能看到本次数据流向预览。
12. 敏感文件和密钥默认脱敏或阻断。
13. Agent 试图突破运行契约会触发审批。

## 17. 最终结论

本产品最终采用 Tauri + React + Rust + Python LangGraph 的桌面端架构。Tauri 和 Rust 负责本地系统能力、Git Worktree、命令执行、安全审批、隐私账本、运行契约和数据持久化；React、TypeScript、Tailwind CSS、shadcn/ui 和 Monaco Editor 负责任务看板、审批流、日志面板、Diff 审查、运行契约面板、记忆控制台和国际化界面；Python、LangGraph、Tree-sitter 和 OpenAI-compatible 模型接口负责 Agent 编排、代码上下文检索、自动修复、偏好蒸馏、Token 预算和模型调用。存储上采用 SQLite + 文件系统产物目录 + Git Worktree + 滚动摘要记忆，不复制完整仓库，不把大日志和截图塞进数据库，以减少本地占用并保持历史审查能力。产品差异化上，以审计级证据包、质量门禁、隐私证明账本、可控记忆、个人喜好蒸馏、运行契约、不降智省 Token、本地低占用、内置 Skills/规则/Hooks、任务胶囊、多模型对战、交付评分和超多国际化作为亮点，避免只停留在普通 AI 代码生成或简单 Agent 执行。

MVP 的核心不是做一个新的代码编辑器，而是先跑通“隔离工作区中的 Agent 编程任务闭环”。只要能够稳定完成创建任务、隔离执行、自动验证、失败修复、Diff 审查和人工合入，产品的基础价值就成立。
