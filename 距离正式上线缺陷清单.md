# CodeMax 距离真正上线缺陷清单

## 1. 文档目的

本文档只记录 CodeMax 从“开发环境可运行”到“可交付给真实用户正式使用”之间仍需关闭的缺陷与任务。

本文档不包含工期、排期、人日、完成日期或甘特图。只有当任务满足对应验收条件并留下可复核证据时，才能关闭缺陷。

## 2. 正式上线判定

“真正上线”不是前端可以打开、测试数量足够或开发机可以运行，而是同时满足以下条件：

1. 用户可以在干净 Windows 环境完成安装、首次启动、配置、真实任务执行、审批、验证、交付、合并、重启恢复、升级和卸载。
2. Agent 的文件读写、命令执行、模型请求、审批、隐私、证据和合并均有不可绕过的安全边界。
3. 应用、Agent、模型、网络或系统异常不会造成静默数据丢失、重复危险操作、越界修改或不可恢复状态。
4. 用户能够理解并控制模型、权限、存储位置、空间占用、日志、隐私、记忆和清理范围。
5. 发布产物可复现、可签名、可升级、可回滚、可诊断，并完成安装态真实 E2E。
6. 默认产品链路不依赖 Demo、Fixture、固定 Todo、固定模型输出、硬编码任务或特制修复指令。

## 3. 优先级与关闭规则

| 等级 | 定义 | 上线规则 |
| --- | --- | --- |
| P0 | 会造成越权、数据破坏、隐私泄露、事务损坏或核心链路不可用 | 未关闭不得发布任何面向真实用户的正式版本 |
| P1 | 正式产品完整性、可靠性、发布工程或用户可控性缺失 | 未关闭不得宣布正式上线 |
| P2 | 不阻断受控灰度，但影响性能、一致性、易用性或长期维护 | 必须进入发布清单并明确验证状态，不得静默遗留 |

关闭任一缺陷必须同时满足：

- 修复代码已经进入拟发布版本；
- 自动化测试通过；
- 对应真实场景验证通过；
- 异常路径和回滚路径通过；
- UI 状态与后端真实状态一致；
- 产生可复核的日志、报告或发布证据；
- 不以“开发机可以运行”代替安装态验收。

## 4. 当前已具备但不能替代上线验收的基线

当前工程已经具备 Python、Rust、契约、UI smoke、release smoke 和桌面构建等自动化验证基础。现有基线可以证明大量模块已可运行，但不能替代以下内容：

- Windows 安装包中的真实完整任务链；
- 崩溃、断电、重启和部分提交后的恢复；
- Windows 文件系统最后一跳的句柄级安全边界；
- 安装、签名、升级、回滚和卸载；
- 长任务、大仓库、大日志和大 Diff 的资源稳定性；
- 真实用户环境中的权限、网络、模型和磁盘故障。

因此，下列缺陷以“是否拥有正式发布所需的完整实现和证据”为关闭标准，而不是以测试总数或页面存在为关闭标准。

# 5. P0：未关闭不得正式发布

## REL-P0-001 Windows 文件编辑最终安全边界

- **所属线路：** A / B / D
- **当前缺口：** 现有路径规范化、多次校验和 Python `pathlib` 防护只能降低风险，仍需证明文件打开、写入、替换、重命名和删除全过程固定在已验证的 Windows 文件句柄上。必须关闭 junction、symlink、mount point、reparse point 及最后一跳目标替换造成的 TOCTOU 窗口。
- **用户风险：** Agent 可能在校验后操作到工作区之外的文件，造成越权读取、覆盖或删除用户数据。
- **修复任务：**
  1. 在 Rust/Win32 层实现 handle-pinned 安全文件操作服务。
  2. 打开路径时拒绝或显式处理 reparse point，并校验最终句柄解析路径仍位于授权根目录。
  3. 写入使用同目录安全临时文件、刷盘和原子替换；替换前后都验证目标身份。
  4. 删除、重命名、覆盖和创建目录统一走同一安全服务，禁止 Python 或前端绕过。
  5. 为 junction 置换、父目录替换、目标文件替换、大小写变体、UNC/长路径、设备路径和并发攻击补充测试。
- **验收条件：** 对抗测试无法使任何读写删除逃逸授权根目录；所有文件修改路径均可追踪到统一安全入口；失败时不留下半写文件或越界副作用。
- **上线门禁：** 阻断。

## REL-P0-002 跨进程文件编辑事务

- **所属线路：** A / C / D
- **当前缺口：** Python Agent、Rust/Tauri、数据库与文件系统之间尚需形成可恢复、可幂等的统一编辑事务，不能只依赖单进程内异常处理或内存状态。
- **用户风险：** 应用或 Agent 中断后可能出现数据库显示成功但文件未完成、文件已修改但事件未落库、回滚只完成一部分等不一致状态。
- **修复任务：**
  1. 定义编辑事务状态机、事务 ID、操作清单、前镜像/备份、提交点和回滚点。
  2. 在修改前持久化事务意图，修改后持久化结果，提交后再发布成功事件。
  3. 让 Rust 文件服务成为实际提交者；Python Agent 和模型不得绕过该服务直接写文件，但可在多轮工具循环中提交受约束的读取、搜索、编辑和命令请求。
  4. 支持重复请求幂等、部分执行恢复、提交重放保护和失败补偿。
  5. 将事务状态关联到 task、审批、Diff、验证和 Proof Pack。
- **验收条件：** 在任意操作步骤强制终止 Python Agent、Tauri 或系统后，重启均能确定恢复、继续或完整回滚；数据库、文件、Diff、事件和 UI 最终一致。
- **上线门禁：** 阻断。

## REL-P0-003 崩溃、重启与危险动作恢复

- **所属线路：** A / B / C / D
- **当前缺口：** 需要完整覆盖 planning、editing、validating、repairing、waiting_approval、delivering、merging 等阶段的异常退出恢复，并区分可安全重试与必须人工确认的动作。
- **用户风险：** 重启后任务可能卡死、重复扣费、重复执行命令、重复删除、重复合并，或错误显示为已完成。
- **修复任务：**
  1. 为全部持久状态定义恢复策略和不可恢复状态的人工入口。
  2. 启动时扫描未完成事务、孤立 Agent、未完成命令、待审批动作、验证中断和合并中断。
  3. 删除、覆盖、外部命令、推送、合并等高风险动作禁止静默自动重放。
  4. 模型调用、命令执行和文件提交使用幂等键或明确的重复执行判定。
  5. UI 展示“已恢复、等待确认、已回滚、需要人工处理”的真实状态与原因。
- **验收条件：** 在每个关键阶段注入进程终止、Agent 断连和系统重启，任务均不会静默重复危险操作，不会丢失可交付证据，并可由用户明确恢复或终止。
- **上线门禁：** 阻断。

## REL-P0-004 安装态真实完整 E2E

- **所属线路：** A / B / C / D
- **当前缺口：** 开发环境测试、Vite build 和 release smoke 不能证明安装后的正式产物已完成真实闭环。
- **用户风险：** 安装后可能出现 Agent 无法启动、资源缺失、数据库路径错误、模型配置不可用、任务无法恢复或打包版本与开发环境行为不一致。
- **修复任务：**
  1. 使用正式安装器在干净 Windows 用户环境安装，不依赖源码目录、开发 Python、Node.js、Rust 工具链或开发环境变量。
  2. 完成首次启动、自检、模型配置与连接测试。
  3. 选择真实 Git 仓库，创建 task、branch、worktree 和 Agent session。
  4. 让真实模型或受控真实兼容端点生成 Todo、读取上下文并修改代码。
  5. 覆盖验证失败、自动修复、审批高风险动作、Diff、Gate、Proof Pack 和本地合并。
  6. 重启应用并恢复任务、日志、Diff、隐私账本、审批与证据。
  7. 检查存储占用并执行安全清理。
  8. 保存完整 E2E 报告、日志、安装器哈希和失败诊断产物。
- **验收条件：** 正式安装产物在无开发依赖的干净环境中连续通过完整用户链路；所有页面数据来自真实 task id 和真实后端状态；失败路径可诊断、可恢复。
- **上线门禁：** 阻断。

## REL-P0-005 审批授权绑定与防重放

- **所属线路：** B / C / A
- **当前缺口：** 高风险动作必须证明审批不是“存在一个 approval id 即可”，而是严格绑定任务、动作、目标、参数、内容摘要、权限范围和有效状态。
- **用户风险：** 已批准的删除、覆盖或命令审批可能被跨任务复用，或在目标内容变化后仍被执行。
- **修复任务：**
  1. 审批记录绑定 task、actor、action、target、arguments digest、content/version digest、scope 和 nonce。
  2. 审批只允许消费一次；拒绝、取消、目标变化、任务变化和契约变化后立即失效。
  3. 执行前在 Rust 权限边界重新校验，不接受仅由前端或 Python Agent 声明的批准状态。
  4. 审批 UI 清楚展示将执行的具体动作、目标、影响范围和不可逆风险。
  5. 增加跨任务复用、参数替换、目标替换、并发消费和重启重放测试。
- **验收条件：** 任意被篡改、复用、过期、已消费或上下文变化的审批均无法执行；拒绝不会形成永久卡死状态；全过程可审计。
- **上线门禁：** 阻断。

## REL-P0-006 隐私与模型请求不可绕过

- **所属线路：** B / A / D
- **当前缺口：** 所有模型请求、上下文拼装、工具返回和重试必须证明统一经过隐私扫描、脱敏、预算和 Privacy Ledger，不能存在旁路调用。
- **用户风险：** 密钥、证书、`.env`、用户目录信息、日志敏感内容或长期记忆可能被发送给外部模型而用户不知情。
- **修复任务：**
  1. 建立唯一模型网关，禁止其他模块直接调用模型端点。
  2. 对 prompt、文件片段、终端输出、工具结果、记忆和重试上下文统一扫描与脱敏。
  3. 对高风险内容执行阻断或明确审批，不允许仅记录不拦截。
  4. Privacy Ledger 记录读取来源、发送范围、脱敏规则、供应商、模型和阻断结果。
  5. 日志、错误事件、遥测和 Proof Pack 使用同一脱敏策略。
  6. 增加编码变体、分片密钥、嵌套 JSON、二进制/证书、路径用户名和异常回显测试。
- **验收条件：** 任何模型请求均能关联到隐私账本和预算记录；敏感测试语料不会以明文出现在网络请求、日志、数据库、事件或证据包中；旁路调用测试失败即阻断发布。
- **上线门禁：** 阻断。

## REL-P0-007 合并与 Git 工作区数据保护

- **所属线路：** A / C
- **当前缺口：** 合并前需要可靠处理目标分支变化、脏工作区、worktree 丢失、冲突、重复合并和部分合并，且不能静默覆盖用户已有修改。
- **用户风险：** 用户未提交的代码可能被覆盖，任务可能合并到错误分支，冲突状态可能被误报为成功。
- **修复任务：**
  1. 合并前重新校验 repository、HEAD、target branch、task branch、worktree 和预览基线。
  2. 目标分支或 Diff 变化后使旧 merge preview、Gate 和审批失效。
  3. 脏工作区默认阻断并提供明确处理选项，不自动 stash、reset 或覆盖。
  4. 合并操作幂等化，记录开始、结果、冲突文件、提交哈希和恢复建议。
  5. 对冲突、用户中途修改、分支删除、worktree 移动和重复点击进行集成测试。
- **验收条件：** 任何基线变化都会触发重新预览与重新确认；冲突不会被标记为成功；用户原有修改不会被静默覆盖或删除。
- **上线门禁：** 阻断。

## REL-P0-008 默认 Agent 主链去 Demo 化

- **所属线路：** A / B / C / D
- **当前缺口：** 正式默认链路必须逐项证明任务、Todo、模型方案、编辑、修复、Diff、交付和合并均来自当前真实任务与真实后端数据，而不是 fixture、固定 task id、固定 Todo、demoDiff、demoDelivery、预制报告或只为测试样本生效的特制指令。
- **用户风险：** 产品在演示路径看似成功，但面对真实仓库和不同任务时无法工作，甚至展示与实际文件状态不一致的成功结果。
- **修复任务：**
  1. 全仓扫描并移除正式默认路径中的 fixture、demo 数据、硬编码 task id、固定 Todo 和预制交付结果。
  2. Todo、工具选择、编辑请求和自动修复必须由当前任务、仓库上下文、历史工具结果和真实模型输出驱动。
  3. 对模型无效输出执行 schema 校验、安全重试、降级或人工介入，禁止直接进入文件写入。
  4. UI 的任务状态、Diff、验证、审批、Proof Pack 和合并结果只能读取当前真实 task id 的持久化数据。
  5. 增加多种仓库、多种任务、模型异常输出和无匹配测试样本的端到端验证。
- **验收条件：** 不同真实任务产生与输入相符的不同 Todo、编辑和修复结果；移除测试 fixture 后正式链路仍可完成；UI 不出现后端未确认的成功、Diff 或交付结论。
- **上线门禁：** 阻断。

## REL-P0-009 Codex 式多轮工具调用主链

- **所属线路：** A / B / C / D
- **当前缺口：** 当前 Agent 即使能够通过固定的 planning、editing、validating 和 repairing 节点完成任务，也不能证明模型具备 Codex 式自主工具循环。产品级内置系统提示词已经定义多轮工具行为，但模型网关、消息协议、工具注册、循环调度、上下文回灌和安装态 E2E 必须形成真实执行链，不能继续以一次性 `TodoPlan` / `EditingPlan` 作为最终能力上限。
- **用户风险：** Agent 只能根据预先注入的有限文件快照生成一次批量编辑，无法在陌生仓库中根据搜索结果继续读取调用方、根据命令输出调整方案、主动补充验证或更换修复策略；产品表面类似 Codex，实际仍是固定阶段的计划执行器。
- **修复任务：**
  1. 扩展统一模型网关和供应商适配层，支持工具定义、工具选择、工具调用结果、`assistant` / `tool` 消息续接、取消、超时和协议错误映射。
  2. 建立 Runtime 权威工具注册表，为仓库搜索、目录与文件读取、结构化编辑或补丁、命令执行、Git 检查、验证、Todo、审批和任务状态提供严格 Schema；每次任务只向模型注入当前真实可用工具。
  3. 实现可持久化的多轮 Agent 循环：模型选择下一项工具，Runtime 校验并执行，真实结果回灌模型，模型继续决定读取、编辑、验证、修复、请求审批或结束，直到进入明确终态。
  4. 固定 planning/editing/repairing 节点和 `EditingPlan` 仅允许作为兼容层或具体工具的数据契约，不得限制整个任务只能执行固定次数、固定顺序的模型交互。
  5. 所有工具调用统一经过 Run Contract、路径隔离、命令策略、审批、隐私扫描、预算、幂等和事件持久化；文件变更仍必须由 Rust 安全文件服务实际提交。
  6. 保存每轮工具调用的请求摘要、参数摘要、结果摘要、状态、耗时、关联事务和上下文来源；敏感原文遵守 Privacy Ledger 与脱敏策略。
  7. 实现上下文压缩、长输出截断、重复读取抑制、循环检测、最大步骤保护和用户取消；达到限制时进入可恢复的人工介入状态，不得伪报完成。
  8. 将 `codemax-programming-assistant.md` 的产品级系统提示词和动态 Runtime Context 接入真实模型链路，移除或降级仍要求“禁止工具调用、每次只返回 JSON”的冲突节点提示词。
  9. 增加多仓库、多语言和陌生任务 E2E，覆盖搜索后追加读取、多文件修改、验证失败后换策略修复、工具不可用、无效工具参数、审批拒绝、网络中断、取消和重启恢复。
  10. 在正式安装包中使用真实模型或受控真实兼容端点完成完整工具轨迹，并保存可复核的工具事件、Diff、验证结果、隐私账本和 Proof Pack。

## REL-P1-016 诊断、日志与隐私安全

# 7. P2：灰度质量与体验缺陷

## REL-P2-001 前端主包体积与按需加载

- **所属线路：** D
- **当前缺口：** 当前桌面主 chunk 约为 `4,104.92 kB`，gzip 后约为 `1,063.31 kB`。Monaco、Diff、大日志和交付聚合需要进一步拆分，避免首页首屏承担无关模块成本。
- **用户风险：** 首次启动慢、内存峰值高，低配置设备交互卡顿。
- **修复任务：**
  1. Monaco、Diff viewer、日志查看器、图表和交付审查按路由或功能懒加载。
  2. 分析重复依赖和不可 tree-shake 模块，设置 bundle budget。
  3. 首页只加载侧栏、composer、项目选择和必要状态。
  4. 测量冷启动、首屏可交互、切页和内存峰值。
- **验收条件：** 构建产物满足明确 bundle budget；首页不预加载 Monaco、大 Diff 或大日志；低配置环境可流畅完成首个操作。
- **上线门禁：** 可受控灰度，但必须记录验证结果。

## REL-P2-002 大仓库、大 Diff 与大日志交互性能

- **所属线路：** A / C / D
- **当前缺口：** 极端规模下的仓库扫描、文件树、搜索、Diff、时间线和终端日志需要分页、虚拟化、取消和渐进呈现。
- **用户风险：** UI 假死、WebView 内存暴涨、任务无法取消。
- **修复任务：**
  1. 文件、日志、事件和 Diff 使用分页、虚拟列表或增量流式加载。
  2. 仓库扫描支持忽略规则、数量上限、进度和取消。
  3. 超大单文件和二进制文件提供降级展示与外部打开选项。
  4. 压测多任务历史和长时间线，避免一次性序列化全部数据。
- **验收条件：** 超大数据场景仍可取消、导航和恢复；UI 不出现不可控冻结；资源占用符合产品预算。
- **上线门禁：** 可受控灰度，但必须有保护阈值。

## REL-P2-003 空状态、错误状态与恢复入口一致性

- **所属线路：** A / B / C / D
- **当前缺口：** 首页、任务线程、设置、隐私、存储、交付和技能页的空状态、失败状态、重试与恢复动作需要统一。
- **用户风险：** 用户不知道系统是否仍在运行、是否已修改文件以及下一步该做什么。
- **修复任务：**
  1. 为加载、空、离线、权限拒绝、后端不可用、数据损坏和部分成功建立统一状态规范。
  2. 每个错误明确说明影响、是否产生副作用和可执行操作。
  3. 禁止仅显示错误码、无限 spinner 或无结果的“重试”。
  4. 保证恢复入口不会清除用户未提交信息。
- **验收条件：** 关键页面错误均能回答“发生了什么、数据是否改变、下一步是什么”；重试和恢复行为可验证。
- **上线门禁：** 可受控灰度，但核心安全错误不得降级处理。

## REL-P2-004 多主题与视觉一致性

- **所属线路：** D
- **当前缺口：** 默认 Mac Minimal 视觉需要与暗色、高对比、紧凑等可切换 UI 保持组件、状态和安全提示一致。
- **用户风险：** 某些主题下文字不可读、危险状态不明显或布局错位。
- **修复任务：**
  1. 使用设计 token 管理颜色、间距、圆角、阴影、字体和状态。
  2. 覆盖首页、任务线程、Inspector、Diff、终端、审批、设置和弹窗。
  3. 主题切换实时生效并持久化，不破坏当前任务状态。
  4. 对中英文、系统缩放和窄屏执行视觉回归。
- **验收条件：** 默认简约 UI 与可选主题均无溢出、低对比或状态缺失；主题切换不触发数据重载或丢失。

- **所属线路：** D / B

  1. 首次启动只引导完成必要设置，并允许稍后配置高级项。
  2. 首页保持简洁，将契约、隐私、存储和证据放入按需展开的 Inspector。

  4. 引导可跳过、可重新打开且不强迫上传数据。
- **验收条件：** 新用户可独立完成首个安全任务；高级信息可发现但不压迫主流程；跳过引导不降低安全默认值。

## REL-P2-006 可选遥测的透明与最小化

- **所属线路：** D / B
- **当前缺口：** 如引入崩溃或使用遥测，必须明确可选、最小化并与隐私账本及日志脱敏保持一致。
- **用户风险：** 用户源码、路径、提示词或身份信息被意外上传。
- **修复任务：**

- **验收条件：** 清理临时数据后仍可打开任务胶囊；历史缺失资源有明确说明；大量任务下搜索和恢复稳定。

# 10. 最终结论

当前 CodeMax 已具备较完整的工程与自动化测试基础，但距离“真正上线”仍以 P0 安全、事务、恢复、审批、隐私、Git 数据保护、Codex 式多轮工具调用主链和安装态完整 E2E 为首要阻断项；P1 则负责补齐发布工程、运行时、存储透明、模式行为、故障恢复、交付可信、国际化、无障碍、资源稳定和用户文档。

在所有 P0、P1 关闭且跨线路最终发布门禁全部通过之前，只能定义为开发验证版或受控测试版，不应对外声明为正式上线版本。
