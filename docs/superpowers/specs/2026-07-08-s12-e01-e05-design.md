# S12 E01-E05 Design

## Scope

This design covers S12-E01 through S12-E05:

- E01: multi-task scheduling, queueing, concurrency limits, and isolation checks.
- E02: Tree-sitter-aware context parsing and relevant context retrieval.
- E03: Playwright-based screenshot artifacts and basic UI checks.
- E04: proposal generation, selection, and regeneration flow.
- E05: Proof Pack, quality gates, override records, delivery score, risk radar, task capsule, and multi-model arena metadata.

The UI stays on the existing simple workspace-first style that the user approved. New UI surfaces are compact panels and cards inside the current task execution page, with Chinese and English i18n keys.

## Architecture

S12 is implemented as a thin enhancement layer over the existing S5/S8/S10 loop. Python owns Agent-side scheduling, parsing, proposal generation, screenshot automation metadata, and model-arena planning objects. Rust owns persisted delivery evidence, quality-gate enforcement, Proof Pack creation, and merge precheck integration. React renders the new information without changing the overall navigation or page shell.

Large artifacts remain on disk under `app-data/tasks/<task-id>/`; SQLite stores only paths, summaries, scores, and indexes. Context retrieval must never default to reading a whole repository. It uses file discovery limits, extension/language filtering, task-query scoring, and parser summaries before any full file is included.

## E01 Multi-Task Parallelism

Add a Python `TaskScheduler` service with configurable concurrency, FIFO queueing, task status snapshots, and per-task state isolation. Tests verify that a failed task frees a slot and does not mutate another task's status.

## E02 Parser And Context Retriever

Add an Agent context package:

- `language_registry.py`: maps mainstream source extensions to language ids and parser names.
- `parser_service.py`: parses source with optional Tree-sitter when available and deterministic lexical fallback when not.
- `retriever.py`: scores files by task query, path, imports, symbol names, and Git/diff hints, returning bounded context snippets.

Mainstream languages include TypeScript, JavaScript, Python, Java, Go, Rust, C, C++, C#, PHP, Ruby, Kotlin, Swift, Dart, Scala, Shell, SQL, HTML, CSS, Vue, Svelte, YAML, TOML, JSON, and Markdown. Parser output contains language, imports, functions, classes, symbols, and parse mode. If Tree-sitter grammar is unavailable, the response explicitly says `parserMode: fallback` instead of pretending.

## E03 Screenshot Validation

Add a lightweight screenshot service that accepts a task id, URL, output directory, viewport, actions, and assertions. It writes screenshot files under task artifacts when Playwright is available, and reports `browserUnavailable` clearly when it is not.

## E04 Proposal Selector

Add proposal models and a deterministic proposal service that generates 2-3 proposals for complex tasks. Each proposal records advantages, drawbacks, risk, impact, estimated effort, and recommended flag. User selection and regeneration are represented in Agent state for later nodes.

## E05 Differentiated Delivery Capabilities

Rust delivery commands become the source of truth for merge-ready evidence:

- Proof Pack directory with manifest JSON, summary Markdown, command/log indexes, diff path, approvals, risks, gates, score, and task capsule.
- Quality gates for test, lint, build, security, approval, diff, and delivery summary.
- Merge gate blocks by default when required gates fail.
- Override flow records reason, gate id, reviewer, and timestamp.
- Delivery score is computed from validation status, risk radar, diff scope, approval status, and delivery explanation completeness.
- Risk radar detects dangerous commands, sensitive files, path escapes, dependency changes, and schema changes.
- Task capsule records replayable summary, decisions, environment hints, verification commands, and evidence index.
- Multi-model arena starts as configuration and proposal comparison metadata; actual multi-provider model calls stay behind the existing model client.

## Testing

The implementation uses TDD:

- Python tests cover scheduler concurrency, parser language coverage, bounded context retrieval, screenshot artifact metadata, and proposal regeneration.
- Rust tests cover Proof Pack manifest creation, gate failure blocking merge, override record persistence, delivery score calculation, and risk radar.
- Frontend contract/build checks cover i18n keys and task page rendering.

## Out Of Scope

S12-E06 and later privacy ledger/run contract/token budget/skills/hooks/i18n theme systems are not implemented here except where E05 writes compatible placeholders into the Proof Pack manifest. Full browser visual QA and real multi-model tournament execution are also left for later stages.
