# CodeMax A/B/C/D 全线完善与上线闭环设计

## 1. 文档目标

本设计用于完成 `最终计划.md` 中 A、B、C、D 四条任务线，并将当前以骨架、展示或部分持久化为主的实现收口为可安装、可恢复、可审计的真实桌面产品。

最终交付不是 Demo，也不能以静态 UI、Fixture、字符串契约检查或硬编码 Agent 行为替代真实功能。所有默认路径必须使用真实仓库、真实任务、真实模型请求、真实运行记录和真实证据。

## 2. 已确认产品决策

### 2.1 UI 方向

- 默认 UI：纯 Mac Minimal。
- 默认首页只展示新任务 composer、项目、模型、思考强度和发送入口。
- 可切换显示方式：浅色、深色、高对比、紧凑模式。
- 紧凑模式可与浅色或深色组合。
- 不引入独立品牌 Minimal 或 Liquid Glass 主题。
- 所有新增文案、状态、错误、占位符和工具提示支持 `zh-CN` 与 `en-US`。
- 应用品牌和安装资源以 `ico/CodeMax.png` 为源。

### 2.2 工作模式

应用提供两个面向用户的工作模式：

1. 日常模式
   - 面向问答、说明、小改动和非技术用户。
   - 非 Git 项目先询问用户是否初始化 Git。
   - 用户同意后初始化 Git，并创建隔离 branch/worktree。
   - 用户拒绝后创建隔离副本，不直接修改原目录。

2. 编程模式
   - 面向真实代码修改、验证、修复和交付。
   - Git 项目默认创建隔离 branch/worktree。
   - 非 Git 项目默认创建隔离副本。
   - 用户可显式授权直接修改原目录，但必须看到风险、影响路径和恢复方式。

任何模式都必须向用户显示实际工作路径、预计磁盘占用、清理影响和永久证据位置。

### 2.3 存储迁移

- 用户可以选择应用数据、worktree/隔离副本和任务产物的存储位置。
- 修改存储位置时使用迁移向导。
- 用户可选择移动已有数据或保留旧数据。
- 迁移前检查空间、权限、目标冲突和数据库状态。
- 迁移失败时回滚配置，不留下半迁移状态。
- 保留旧数据时，历史任务继续记录原始位置，并在 UI 中明确标识。

## 3. 当前基线与主要缺口

当前工作区包含大量未提交的 A/B/D/UI 联调改动。现有实现提供了真实任务入口、部分隐私与证据表、设置页、存储统计、清理、自检和 Skills 页面，但尚未满足最终上线门禁。

主要阻断包括：

- Rust/Tauri 当前存在 `CommandExecutionError::InvalidPurpose` 未覆盖导致的编译失败。
- Python Agent 仍以固定 Todo、计划文件和 `CODEMAX_REPAIR` 特制输出为主要编辑/修复链路。
- 非 Git 目录直接修改会破坏多任务隔离。
- `target_branch` 未冻结持久化，外部分支切换可能改变合并目标。
- 模型请求没有统一经过隐私、预算和上下文网关。
- Run Contract 对命令、网络、路径和预算的约束没有完整执行。
- 桌面 SQLite 记忆与 Python `memory.json` 构成双数据源。
- Proof Pack 使用固定目录，旧证据可能被覆盖或变成陈旧索引。
- Proof Pack 脱敏覆盖不足，合并记录不会进入既有证据版本。
- Hooks 和 Model Arena 目前主要是记录壳，不是真实执行引擎。
- D 线缺少真实模型连通、可选存储位置、Python Agent 随包分发、安装态 E2E 和完整发布文档。
- 前端构建主包约 4 MB，Monaco 与任务详情未按路由懒加载。

## 4. 总体架构

系统分为六个边界明确的模块。

### 4.1 React 桌面 UI

职责：

- Mac Minimal 首页、侧栏、任务线程和设置。
- 日常/编程模式选择。
- 隐私预览、Run Contract、审批和交付审查。
- 存储位置、迁移、占用和清理展示。
- 双语、主题、高对比、紧凑模式和减少动态效果。

约束：

- 不直接读取文件、执行命令或调用模型供应商。
- 首页不加载大日志、Diff、Monaco 或交付审查聚合数据。
- 所有错误通过稳定错误码映射为本地化文案。

### 4.2 Rust/Tauri 编排核心

职责：

- 仓库检查、branch/worktree、隔离副本和任务状态。
- 命令执行、日志、取消、审批和契约执行。
- SQLite、文件产物、存储迁移和清理保护。
- Python Agent 生命周期和 IPC 安全边界。
- 合并前门禁与最终合并。

约束：

- 任务创建必须具备事务式补偿逻辑。
- 所有路径在进入文件系统前规范化并验证边界。
- 高风险行为必须由 Rust 层阻断，不能只依赖 UI 提示。

### 4.3 Python Agent

职责：

- 根据真实任务生成 Todo。
- 选择上下文、调用模型、生成编辑方案并应用修改。
- 分析真实验证日志并执行自动修复。
- 生成候选偏好，不直接写入长期记忆。

约束：

- 不使用固定 Todo 或特制验证输出替代推理。
- 不直接旁路模型网关。
- 不维护与桌面应用冲突的长期记忆数据源。

### 4.4 模型与上下文网关

职责：

- 统一 OpenAI-compatible 模型调用。
- 上下文分层、预算估算和来源记录。
- 发送前扫描、脱敏或阻断。
- 记录真实供应商、模型、usage、延迟和失败。
- 执行 Run Contract 的模型、网络和 Token 约束。

所有 Agent、Model Arena 和连接测试必须复用该网关。

### 4.5 交付证据核心

职责：

- 版本化 Proof Pack。
- Quality Gate、Delivery Score 和 Risk Radar。
- Task Capsule、Rules、Hooks 和 Model Arena 决策。
- 合并前唯一决策状态。

证据版本必须绑定输入快照、内容哈希和生成时间。任务状态、Diff、验证或审批变化后，旧版本标记为陈旧但不得覆盖或删除。

### 4.6 存储与发布层

职责：

- 可选存储位置和迁移向导。
- 临时数据清理与永久证据保护。
- Python Agent 运行时随包分发。
- Tauri Windows 安装器、启动自检和安装态 E2E。
- 用户文档、发布说明和 smoke 报告。

## 5. 共享契约

### 5.1 契约原则

- 以 `contracts/ipc.schema.json` 为单一公共契约来源。
- 关键对象不得使用无约束的 `additionalProperties: true` 空壳。
- Rust、TypeScript 和 Python 类型必须通过自动契约测试对齐。
- IPC command 枚举、请求字段、响应字段和可空语义必须一致。
- 数据库迁移与代码引用必须在同一提交中交付。

### 5.2 必须冻结的核心字段

- Task：`task_id`、`repository_id`、`worktree_path`、`task_branch`、`target_branch`、`task_status`。
- Agent：阶段、Todo、事件、验证轮次、修复轮次和人工介入状态。
- Run Contract：模式、模型、强度、权限、网络、路径、命令、预算、语言、记忆范围和越界策略。
- Privacy：实际读取、实际发送、供应商、记忆、脱敏、阻断和上下文来源。
- Delivery：Diff、验证、审批、隐私、契约、预算、Gate、Score、Risk、Proof Pack 和 merge preview。
- D 线：应用设置、模型设置、存储占用、迁移状态、清理结果、启动健康、安装产物和 smoke 报告。

## 6. 任务数据流

1. 用户选择工作模式、项目和任务描述。
2. Rust 检查 Git 状态、路径权限、存储空间、模型配置和 Agent 状态。
3. Git 项目创建独立 branch/worktree；非 Git 项目按已确认的模式策略处理。
4. 系统生成隐私预览与 Run Contract。
5. 用户确认后创建任务、Agent session 和初始证据快照。
6. Python Agent 通过模型网关生成 Todo。
7. Context Budgeter 按最近消息、摘要、记忆、文件片段和工具结果分层加载。
8. 每次读取、发送、命令、审批、记忆和文件修改写入事件与 Privacy Ledger。
9. 验证失败后根据真实日志修复；达到上限后进入人工介入。
10. 完成后生成 Diff、验证汇总、Gate、Score、Risk 和版本化 Proof Pack。
11. 用户在交付审查页做最终合并决策。
12. 合并后追加 merge record，并生成新的最终证据版本。
13. 应用重启后从持久化数据恢复任务、Diff、日志、审批、证据和 Task Capsule。

## 7. A 线完成设计

### 7.1 真实 Agent 主链

- 模型客户端接入 Agent graph。
- Todo 根据任务和仓库上下文生成。
- 编辑阶段生成结构化文件修改并应用到隔离工作区。
- 验证失败后以真实 stdout/stderr 和 Diff 为输入生成修复。
- 移除生产链路对 `CODEMAX_REPAIR` 特制输出的依赖。

### 7.2 隔离与恢复

- Git 任务始终使用独立 branch/worktree。
- 子目录通过 Git 根目录识别，不误判为非 Git。
- 非 Git 隔离副本记录源路径、工作路径、创建时间和预计占用。
- 直接修改原目录必须记录明确授权。
- `target_branch` 在任务创建时持久化冻结。
- 创建失败回滚任务记录、分支、worktree、副本和 Agent checkpoint。

### 7.3 历史与 Diff

- 任务列表使用分页或游标，不固定截断为 50 条。
- Diff、日志、事件和验证轮次从持久化数据恢复。
- 多任务在同仓库中并行时互不污染。

## 8. B 线完成设计

### 8.1 隐私与契约

- 模型发送、文件读取、命令输出和记忆使用统一经过隐私服务。
- 扫描覆盖常见供应商 Token、JWT、Bearer、URL 凭据、证书、私钥和数据库连接串。
- Run Contract 对命令、路径、网络、权限、单次预算和总预算真实生效。
- 越界请求进入可恢复审批流。

### 8.2 Token 与上下文

- 使用供应商真实 usage；缺失时采用明确标记的估算值。
- Context Budgeter 接入任务主链，不默认读取整仓。
- 关键上下文来源进入 Privacy Ledger 与 Proof Pack。
- 超预算时按契约阻断、降级或请求审批。

### 8.3 记忆、偏好与画像

- SQLite 成为长期记忆的唯一来源。
- Python Agent 通过受控 API 获取任务可用记忆。
- 支持查看、编辑、删除、停用、作用域和最近使用时间。
- 候选偏好只有在用户接受或编辑后接受时写入长期记忆。
- 预置默认、安全优先、严格测试、低成本、快速原型和深度重构画像。
- 画像同时影响契约与实际执行行为。

## 9. C 线完成设计

### 9.1 不可变证据

- Proof Pack 路径包含版本 ID，不使用固定可覆盖目录。
- 清单记录文件哈希、输入快照哈希和生成状态。
- Diff、验证、审批、隐私或契约变化会使旧版本标记为陈旧。
- 合并后生成包含最终 merge record 的新版本。
- 所有文本和文件在进入证据包前执行统一敏感扫描。

### 9.2 Gate、Score 与 Risk

- Gate 使用当前有效验证轮次，不因已解释或已修复的历史失败永久阻塞。
- 审批、Risk 和契约突破具备 pending、resolved、overridden 等可消解状态。
- Score 覆盖验证、Diff、修复、风险、审批、契约、隐私和用户反馈。
- Score 和 Risk 在输入变化后自动失效并重新计算。

### 9.3 Rules、Hooks 与 Model Arena

- 内置 Bug 修复、补测试、安全检查和发布前检查规则。
- Hooks 在命令前、写文件前、验证后和合并前真实执行。
- Hook 扩权或执行命令必须获得已批准的审批，不能只检查审批 ID 是否存在。
- Model Arena 至少调用两个真实模型配置生成方案。
- UI 比较影响范围、风险和预计成本，用户选择后才允许执行。

## 10. D 线完成设计

### 10.1 设置与模型

- OpenAI-compatible Base URL、API Key、模型名和输出语言可配置。
- API Key 使用系统安全存储并始终脱敏显示。
- 连接测试通过统一模型网关执行最小真实请求，并返回脱敏结果。
- 日常/编程模式、思考强度和主题设置持久化。

### 10.2 存储治理

- 展示数据库、worktree、隔离副本、产物、日志、截图、临时上下文和永久证据。
- 支持存储迁移向导与失败回滚。
- 清理先预览，再确认执行。
- Proof Pack、Task Capsule、审批和合并记录默认不可删除。

### 10.3 打包与发布

- Python Agent 和必要 Python 运行时或独立可执行产物随 Tauri 安装包分发。
- 启动自检验证数据库、路径、模型、Agent 和资源文件。
- 窗口、任务栏和安装器使用 CodeMax 官方图标。
- 生成 Windows MSI/NSIS 安装器并在干净环境验证。
- 交付安装说明、配置说明、隐私/记忆/存储说明、已知问题和发布说明。

## 11. UI 设计

### 11.1 首页与侧栏

- 304px 浅灰侧栏，窄屏时折叠为图标导航。
- 首页中央为“我们该做什么？”和大 composer。
- composer 提供添加、模式、模型/思考强度、项目和发送操作。
- 不在首页展示日志、Diff、Gate、Score、Proof Pack 或存储图表。

### 11.2 任务线程

- 主区按时间展示用户输入、Todo、Agent 阶段、命令、验证、修复和审批。
- 命令输出和 Diff 默认折叠，并支持增量加载。
- 右侧检查器默认折叠，包含契约、模型、权限、存储、隐私和证据。

### 11.3 交付审查

- 合并前唯一决策页面。
- 同时展示 Diff、验证、审批、隐私、契约、预算、Gate、Score、Risk、Proof Pack 和 merge preview。
- Gate 失败时合并操作不可用；覆盖必须输入原因并进入证据。

### 11.4 设置

- 左侧分类、右侧行式设置。
- 覆盖常规、外观、模型、思考强度、个性化、权限、技能、Hooks、Git、环境、工作树、存储、语言、启动和关于。
- 使用 Lucide 图标、工具提示、键盘操作和明确的焦点状态。
- 状态不能只依赖颜色表达。

## 12. 错误处理与安全

- 使用稳定错误码和本地化提示。
- 不向前端返回底层秘密、完整环境变量或供应商原始错误体。
- Agent、模型和命令中断时保存检查点。
- 分支漂移、源分支异常、合并冲突和证据陈旧时阻断合并。
- 用户可以重试、恢复、调整契约或人工接管。
- 已拒绝或已修复问题通过状态机消解，不形成永久错误阻塞。

## 13. 性能与资源要求

- 首页首屏不加载 Monaco、Diff、大日志或交付聚合。
- 任务详情、设置、Skills 和 Monaco 按路由/功能动态导入。
- 日志与事件使用分页或增量读取。
- 技能扫描先返回来源，再懒加载描述。
- 存储统计按目录流式遍历，只保留聚合计数。
- 记录前端包体、启动耗时、空闲内存、任务内存和存储扫描耗时。
- 动效遵循 `prefers-reduced-motion`。

## 14. 实施顺序

采用风险优先的纵向闭环：

1. 修复当前编译错误并冻结共享契约。
2. 完成真实模型驱动的任务、编辑、验证和修复闭环。
3. 完成隐私、契约、上下文、预算和单一记忆源闭环。
4. 完成不可变证据、Gate、Score、Risk、Hooks 和 Model Arena。
5. 按 Mac Minimal 收口首页、任务线程、交付审查、设置、主题和国际化。
6. 完成存储迁移、资源优化、Agent 打包、安装器和安装态 E2E。

每一步都必须产生可运行、可测试的纵向结果，不能等到最后一次性联调。

## 15. 测试与验收

### 15.1 自动化测试

- Rust/TypeScript/Python/JSON Schema 三方契约测试。
- Rust 单元与集成测试覆盖任务事务、Git 隔离、审批、迁移、证据和合并。
- Python 测试覆盖真实模型 mock、上下文、编辑、修复、记忆和偏好候选。
- 前端测试覆盖模式、主题、i18n、存储迁移和交付决策。
- Playwright 覆盖桌面与窄屏截图、文本溢出、重叠和可操作性。

### 15.2 关键 E2E

- 同仓库双任务隔离。
- 应用重启后恢复 Diff、日志、事件和证据。
- 真实模型请求经过脱敏、账本和预算。
- 删除或停用记忆后，后续任务不再使用。
- 两次 Proof Pack 生成不覆盖，输入变化后旧包失效。
- 四个 Hook 生命周期真实拦截。
- 两模型 Arena 比较并记录选择。
- 存储迁移成功、保留旧数据和失败回滚。

### 15.3 安装态验收

在 Windows 安装包中完成：

1. 安装和首次启动。
2. 启动自检。
3. 配置并测试模型。
4. 选择真实仓库。
5. 创建任务和独立 worktree。
6. Agent 生成 Todo 并修改代码。
7. 验证失败后自动修复。
8. 审批高风险动作。
9. 打开交付审查并生成 Proof Pack。
10. Gate 通过后合并。
11. 重启应用并恢复历史任务。
12. 查看存储占用并清理临时数据。

## 16. 上线门禁

以下任一情况存在时不得上线：

- Rust、Python、前端或契约测试失败。
- 默认链路仍依赖 Fixture、固定 Todo、固定模型方案或特制修复指令。
- 模型请求可绕过隐私与预算网关。
- 非 Git 项目可在未授权情况下直接修改原目录。
- Proof Pack 可覆盖、陈旧或包含敏感明文。
- Gate、审批、Risk 或契约突破无法正确消解。
- Hooks 或 Model Arena 只有记录壳。
- 存储位置、占用、迁移或清理范围对用户不透明。
- Python Agent 未随安装包可靠启动。
