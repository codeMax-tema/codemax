# CodeMax 最终上线需求文档（v1.0）

## 1. 文档目标

本文档用于定义 `CodeMax` 的最终上线版本需求呀，目标不是继续做概念规划，而是把现有代码能力、尚未补齐的产品能力、四人并行开发方式、统一验收和上线门禁一次性写清楚。  
开发团队按本文档完成实现后，只需要执行本文档定义的“最终验证清单”并全部通过，即可进入打包上线流程哦。

---

## 2. 产品定义

### 2.1 产品名称

CodeMax

### 2.2 产品定位

CodeMax 不是传统 IDE，也不是单纯的聊天式代码助手呀。  
它是一个本地优先、可审查、可审批、可交付、可回溯的编程智能体工作台，用来把“自然语言任务 -> Agent 执行 -> 验证修复 -> 审查合并”变成一个可控的产品闭环哦。

### 2.3 最终上线版本一句话定义

一个可以在本地真实仓库中创建任务、隔离执行、自动验证修复、统一审查交付、经人工确认后本地合并，并具备隐私、记忆、运行契约、质量门禁和打包交付能力的桌面端编程智能体产品呀。

---

## 3. 当前代码基线与上线判断

### 3.1 已有能力基线

基于现有代码，以下能力已经具备基础实现或核心内核呀：

1. 本地 Git 仓库识别与 `git worktree` 隔离。
2. 本地命令执行、日志采集、取消执行、安全拦截。
3. Diff 生成、交付报告、审批中心、合并核心流程。
4. Python Agent 状态机主链路和基础修复闭环。
5. SQLite 数据层、任务模型、记忆模型、质量门禁/证据包/交付评分表结构。
6. i18n、主题/紧凑模式、模型配置安全存储。
7. S11 级 MVP 验收主链路测试。

### 3.2 当前不能直接上线的根因

当前项目离“可打包上线产品”最大的差距不在后端内核，而在产品化闭环还没完全接通呀：

1. 桌面端任务创建、任务列表、任务详情仍有假数据/演示回退逻辑。
2. Agent 运行状态、日志、修复轮次、审批结果的统一可视化还不完整。
3. 交付审查页还没有真正统一成上线级产品入口。
4. Proof Pack、Quality Gate、Delivery Score 等亮点能力存在数据底座，但尚未完整产品化。
5. Privacy Ledger、Run Contract、Token Budget、Memory Cockpit 还没有形成完整的用户闭环。
6. 打包、首次启动、自检、存储治理、上线 smoke test 还没有收口成可交付标准。

### 3.3 上线版本结论

最终上线版本以“编程智能体产品化能力”作为目标，不追求完整 IDE 化呀。  
只要完成本文档定义的范围，并通过最终验证清单，即视为 `v1.0` 可上线。

## 4. 上线版本目标

### 4.1 必达目标

1. 用户可选择本地 Git 仓库并创建真实任务。
2. 每个任务必须运行在独立 worktree 中。
3. Agent 必须先规划 Todo，再修改代码，再执行验证。
4. 验证失败后必须进入自动修复闭环，且轮次受控。
5. 用户必须能看到实时日志、当前阶段、失败原因和审批结果。
6. 合并前必须有统一交付审查页。
7. 高风险操作必须审批，不允许静默放权。
8. 最终交付必须包含 Diff、测试结果、风险解释、质量门禁、交付评分。
9. 用户必须能看见记忆、隐私、运行契约、预算等影响 Agent 决策的关键控制项。
10. 完成统一验证后可直接打包并上线。

### 4.2 非目标

以下内容不属于 `v1.0` 上线必需项呀：

1. 完整 IDE 编辑器体验。
2. 团队账号体系与云端协作。
3. 自动创建 PR、自动推送远程仓库。
4. 插件市场。
5. 跨多仓库统一编排。

## 5. 用户与使用场景

### 5.1 目标用户

1. 个人开发者：希望把修 Bug、补测试、局部重构交给 Agent 执行呀。
2. 技术负责人：希望能同时管理多个任务，并对交付结果有审查和门禁。
3. QA / 审查者：希望直接查看验证、日志、Diff、风险和合并结果。

### 5.2 核心场景

1. 修复本地仓库中的真实 Bug。
2. 为现有模块补充测试。
3. 在隔离工作区中进行局部重构。
4. 在高风险命令执行前进行人工审批。
5. 在交付页统一决定是否允许合并。

## 6. 最终上线版本的范围定义

### 6.1 范围原则

上线版本必须优先保证以下四件事呀：

1. 真任务，不是 demo。
2. 真审查，不是日志堆砌。
3. 真可控，不是黑箱 Agent。
4. 真可交付，不是“看起来能跑”。

### 6.2 最终上线版本包含的能力域

1. 仓库与任务主链。
2. Agent 执行与可观测。
3. 审批、安全与合并。
4. 交付审查与质量门禁。
5. 记忆、隐私、运行契约、预算控制。
6. 设置、模型、存储、国际化、主题。
7. 打包、安装、自检、上线 smoke test。

## 7. 产品原则

### 7.1 用户优先

任何设计都必须优先回答“用户是否知道发生了什么、为什么发生、下一步能做什么”呀。

### 7.2 本地优先

数据、日志、Diff、截图、审查记录、记忆和配置优先保存在本地，路径透明、可控、可清理哦。

### 7.3 审计优先

不允许只有最终结果没有过程证据。  
每个任务必须可追溯输入、执行、验证、审批、合并和清理行为呀。

### 7.4 安全优先

高风险命令、越界路径、敏感数据、预算超限、契约突破都必须有显式拦截或审批哦。

## 8. 总体架构与边界

### 8.1 技术架构

1. `Tauri + Rust`：系统边界层，负责文件系统、Git、进程、审批、持久化、事件流、打包。
2. `React + TypeScript`：产品操作台，负责任务、日志、审批、审查、设置、亮点能力 UI。
3. `Python + LangGraph`：Agent 编排层，负责规划、修改、验证、修复、记忆提取、模型调用。
4. `SQLite + 文件系统产物目录`：结构化索引与大文件分层存储。

### 8.2 模块边界

1. Rust 不直接负责复杂业务 UI。
2. React 不直接越过 Rust 操作系统资源。
3. Python 不直接操作主工作区，只能经受控路径和受控命令执行。
4. 大日志、大截图、大 Diff 不写入 SQLite，只记录路径和摘要。

## 9. 任务主链需求

### 9.1 仓库选择

1. 用户可以选择本地 Git 仓库。
2. 系统展示仓库路径、当前分支、脏状态、最近使用记录。
3. 非 Git 仓库必须明确拒绝。
4. 脏工作区必须提醒用户，但不强制禁止创建任务。

### 9.2 新建任务

1. 用户可输入自然语言任务描述。
2. 用户可选择任务类型：修 Bug、补测试、重构、自定义。
3. 用户可选择运行模式、模型、语言、预算、验证策略。
4. 创建任务后系统必须真实完成：
   - 创建 task 记录。
   - 创建 task branch。
   - 创建 worktree。
   - 创建 agent 会话。
   - 进入任务详情页。

### 9.3 任务列表

1. 列表必须来自真实数据库，不允许使用 fixture。
2. 支持状态筛选：进行中、等待审批、需人工介入、已完成、已合并、失败、已取消。
3. 支持按仓库筛选和最近更新时间排序。
4. 支持恢复历史任务详情。

### 9.4 任务详情

任务详情页必须成为产品主工作台呀，至少包含：

1. 任务概览。
2. Todo 看板。
3. 运行时间线。
4. 实时日志面板。
5. 验证结果。
6. Diff 预览。
7. 审批记录。
8. 交付审查入口。
9. 亮点能力入口：Proof Pack、Gate、Score、Privacy、Memory、Contract、Budget。

---

## 10. Agent 执行与可观测需求

### 10.1 执行状态机

任务必须按以下状态流转呀：

1. `queued`
2. `planning`
3. `editing`
4. `validating`
5. `repairing`
6. `awaitingApproval`
7. `awaitingReview`
8. `readyToMerge`
9. `merged`
10. `needsIntervention`
11. `failed`
12. `cancelled`

### 10.2 Todo 看板

1. Agent 接到任务后必须先生成 Todo List。
2. 每个 Todo 包含标题、说明、状态、开始时间、完成时间。
3. Todo 状态实时更新到前端。
4. 用户可查看每个 Todo 对应的日志和说明。

### 10.3 日志与运行面板

1. 日志必须实时滚动展示。
2. 大日志必须分页或增量加载。
3. 每条命令必须记录：
   - 命令内容
   - 工作目录
   - 开始/结束时间
   - 退出码
   - stdout/stderr 路径
4. 用户可取消长时间运行命令。
5. 最近一次失败原因必须提炼为可读摘要。

### 10.4 自动修复闭环

1. 验证失败后 Agent 必须分析失败原因。
2. 自动修复默认最多 5 轮。
3. 每轮修复都必须可见：
   - 失败命令
   - 错误摘要
   - 修复意图
   - 新的验证结果
4. 超出最大轮次后任务进入 `needsIntervention`。

---

## 11. 审批、安全与合并需求

### 11.1 审批中心

以下行为必须统一进入审批中心呀：

1. 删除文件或目录。
2. 修改数据库 Schema。
3. 安装、升级、删除依赖。
4. 执行危险命令。
5. 越出任务工作区读写。
6. 合并主分支。
7. 运行契约突破。
8. Hook 扩权。

### 11.2 审批交互

审批卡片必须展示：

1. 动作内容。
2. 执行目录。
3. 风险等级。
4. 影响范围。
5. Agent 理由。
6. 建议结果。

用户必须可选择：

1. 允许
2. 拒绝
3. 要求修改

### 11.3 合并前门禁

合并前必须同时满足：

1. 验证通过。
2. 最终 Diff 非空。
3. 目标工作区干净。
4. 审批项已处理完毕。
5. Quality Gate 通过，或用户显式强制覆盖且记录原因。

### 11.4 本地合并

1. 合并仅允许本地执行，不默认推送远程。
2. 合并前必须二次确认。
3. 冲突时必须自动 `merge --abort`。
4. 冲突必须返回：
   - 冲突文件列表
   - Git 原始原因摘要
   - 合并记录路径
5. 合并成功后必须生成 merge record。

## 12. 交付审查需求

### 12.1 统一交付审查页

交付审查页是上线版本的核心产品页面呀，必须统一展示：

1. 最终 Diff。
2. 验证命令与结果。
3. 测试报告。
4. 失败与修复轮次摘要。
5. 审批记录。
6. Quality Gate 状态。
7. Delivery Score。
8. 风险解释。
9. Proof Pack 导出入口。
10. Privacy Ledger 摘要。
11. Run Contract 摘要。
12. Token Budget 摘要。

### 12.2 Proof Pack

每个完成任务必须可生成 Proof Pack，内容包括：

1. 任务描述。
2. Agent 计划和 Todo。
3. 关键命令与日志摘要。
4. 验证结果。
5. 最终 Diff。
6. 审批记录。
7. 风险说明。
8. 合并结果。
9. 交付总结。

### 12.3 Quality Gate

Quality Gate 必须至少包含：

1. 验证命令全部通过。
2. 无未处理高风险审批。
3. 无未解释的失败轮次。
4. 敏感信息扫描通过。
5. 最终交付产物完整。

### 12.4 Delivery Score

Delivery Score 必须按规则给出分值和解释呀，至少考虑：

1. 验证覆盖度。
2. Diff 规模。
3. 修复轮次。
4. 风险等级。
5. 审批强度。
6. 人工覆盖情况。

## 13. 记忆、隐私、运行契约与预算需求

### 13.1 Memory Cockpit

1. 用户可以查看长期记忆。
2. 用户可以编辑、删除、停用长期记忆。
3. 记忆必须带来源、作用域和最近使用时间。
4. 候选偏好进入长期记忆前必须经用户确认。

### 13.2 Preference Distiller

系统可以从用户反馈中提炼候选偏好，但不能静默写入长期记忆呀。  
候选偏好至少要支持：

1. 接受
2. 编辑后接受
3. 忽略
4. 永不记录同类偏好

### 13.3 Privacy Ledger

每次任务都必须生成 Privacy Ledger，记录：

1. 读取了哪些本地文件。
2. 哪些内容进入模型上下文。
3. 哪些内容被脱敏。
4. 哪些内容被阻止发送。
5. 使用了哪些长期记忆。
6. 调用了哪个模型供应商。

### 13.4 Run Contract

任务启动前必须生成并展示 Run Contract，至少包含：

1. 运行模式。
2. 选用模型。
3. 推理强度。
4. 权限等级。
5. 网络策略。
6. 允许路径。
7. 允许命令范围。
8. 验证命令。
9. 语言。
10. Token 预算。
11. Memory scope。

### 13.5 Token Budget 与 Context Budgeter

1. 任务启动前提供预算预估。
2. 任务执行中记录关键模型调用的 token 使用。
3. 上下文加载必须分层，不能默认吃整仓库。
4. 风险升高或验证失败时允许按规则追加上下文。
5. 用户可以看到本次任务大致用了哪些上下文来源。

## 14. 模型、设置、存储与国际化需求

### 14.1 模型配置

1. 支持 OpenAI-compatible 模型配置。
2. Base URL、API Key、模型名可保存。
3. API Key 必须安全存储和脱敏展示。
4. 支持测试连接。

### 14.2 存储治理

用户必须能在设置页看到并控制以下内容呀：

1. 数据库存储路径。
2. worktree 根目录。
3. 任务产物目录。
4. 日志保留时间。
5. 截图保留时间。
6. 临时上下文保留时间。
7. 一键清理策略。
8. 当前磁盘占用统计。

### 14.3 国际化

1. UI 文案必须支持中英文切换。
2. Agent 输出语言必须可配置。
3. Proof Pack 与交付总结语言必须可配置。
4. 缺失翻译必须回退到默认语言，不能让页面崩掉。

### 14.4 主题与外观

上线版本采用当前已确认的简约工作台风格呀。  
最终版本必须支持：

1. 默认简约主题。
2. 紧凑模式。
3. 高对比模式。
4. 深浅主题切换。

说明：新增大风格 UI 变体需要单独设计确认，不属于本次直接上线必需项哦。

### 14.5 图标与品牌

1. 应用图标、安装包图标、任务栏图标、关于页图标统一使用 `D:\codemax\ico\CodeMax.png` 转换后的正式资源。
2. 不得私自替换品牌主图标。

## 15. 非功能需求

### 15.1 性能

1. 任务列表打开时间不超过 1 秒。
2. 任务详情首屏不超过 2 秒。
3. 日志必须增量加载。
4. 大 Diff 支持折叠和按需展开。
5. 不能一次性把整仓库和全部日志读入内存。

### 15.2 稳定性

1. 单个任务失败不得影响其他任务。
2. 应用重启后任务历史必须可恢复。
3. 应用重启后日志、审批、交付产物必须可恢复。
4. 清理任务后永久证据不能丢失。

### 15.3 安全

1. 敏感信息不得写入普通日志和导出文件。
2. Agent 默认只能操作任务 worktree。
3. 高风险动作必须审批。
4. 黑名单命令必须默认禁止。

### 15.4 资源占用

1. SQLite 只存结构化信息与索引。
2. 大文件必须落盘到产物目录。
3. 保留策略必须可调。
4. 清理策略必须可见且可手动执行。

## 17. 四人并行开发方案

本节是上线版研发组织的核心呀。四个人必须按“边界清楚、尽量并行、最后统一联调”的方式推进。

### 17.1 角色分工总览

1. A：桌面端任务主链负责人
2. B：Agent 可观测与审批负责人
3. C：交付审查与亮点能力负责人
4. D：平台稳定性、打包、设置与验收负责人

### 17.2 A 负责人：任务主链产品化

负责范围：

1. 仓库选择真实化。
2. 新建任务真实化。
3. 任务列表去 fixture。
4. 任务详情加载真实数据。
5. 任务状态流转、导航、恢复。

交付物：

1. 真任务创建入口。
2. 真任务列表和筛选。
3. 真任务详情数据流。
4. 与 Rust/Python 的任务状态联通。

验收标准：

1. 新建任务不再依赖假 task id。
2. 刷新应用后任务仍可恢复。
3. 列表、详情、状态全部来自真实存储。

依赖：

1. Rust 已有 task/worktree 命令。

### 17.3 B 负责人：Agent 可观测与审批闭环

交付物：

验收标准：

依赖：

1. Rust 命令执行与审批命令。

### 17.4 C 负责人：交付审查与亮点能力

负责范围：

1. 统一交付审查页。
2. Diff、测试、审批、Gate、Score 汇总。
3. Proof Pack 导出。
4. Privacy Ledger、Run Contract、Token Budget 摘要 UI。
5. Delivery Score、Risk Radar 展示。

交付物：

1. 上线级交付审查页。
2. Proof Pack 导出能力。
3. Quality Gate 与 Delivery Score 前后端接通。
4. 亮点能力入口与摘要卡片。

验收标准：

### 17.5 D 负责人：设置、稳定性、打包与最终验收

### 17.6 并行边界约束

2. B 只负责运行态与审批体验，不重做任务创建。
3. C 只负责交付页和亮点产品化，不介入底层存储治理。
4. D 只负责设置、资源治理、打包和统一验收，不随意改主链业务。

## 19. 最终上线验收标准

### 19.1 功能验收

1. 可以选择本地 Git 仓库。
2. 可以创建真实任务。
3. 可以创建真实 worktree。
4. Agent 可以生成 Todo。
5. Agent 可以修改代码并执行验证。
6. 验证失败后能自动修复。
7. 用户能看到日志、状态、审批和失败原因。
8. 交付审查页可统一查看全部关键结果。
9. Quality Gate 与 Delivery Score 可用。
10. 用户确认后可本地合并。

### 19.2 安全验收

1. Agent 不直接污染主工作区。
2. 危险命令必触发审批。
3. 拒绝审批后命令不执行。
4. API Key 不进入普通日志。
5. 敏感文件默认脱敏或阻止。
6. Run Contract 突破会被拦截或审批。

### 19.3 亮点验收

1. 每个完成任务可导出 Proof Pack。
2. 每个任务有 Quality Gate 结果。
3. 每个任务有 Delivery Score。
4. 每个任务可查看 Privacy Ledger。
5. 用户可管理长期记忆。
6. 任务详情可查看 Run Contract 和 Token Budget 摘要。

### 19.4 资源与稳定性验收

1. 数据路径可见。
2. 存储占用可见。
3. 清理策略可执行。
4. 重启后历史任务可恢复。
5. 清理后永久证据仍保留。

### 19.5 打包验收

1. 生成 Windows 安装包。
2. 安装后首次启动成功。
3. 图标、应用名、存储路径正常。
4. 安装后的应用能完成一次完整真实任务闭环。

## 20. 最终验证清单

本节是“开发完成后一步收口”的关键呀。  
只要四人并行开发全部完成，就按以下顺序跑最终验证；全部通过即可上线。

### 20.1 自动化验证

必须至少执行以下验证：

1. 前端类型检查与构建通过。
2. Rust 测试通过。
3. Python Agent 测试通过。
4. S11 主链路验收通过。
5. 新增上线版验收脚本通过。
6. 交付页、审批、设置、记忆、存储相关关键测试通过。

### 20.2 手工 smoke 验证

必须由一人完整手工跑以下闭环呀：

1. 选择真实本地仓库。
2. 创建“修 Bug”任务。
3. 系统创建 worktree。
4. Agent 生成 Todo 并执行。
5. 至少发生一次验证失败并完成自动修复。
6. 审批中心处理一次高风险动作。
7. 打开统一交付审查页。
8. 查看 Gate、Score、Proof Pack、Privacy、Contract、Budget。
9. 执行本地合并。
10. 重启应用后查看历史记录和证据。

### 20.3 安全 smoke 验证

1. 故意触发危险命令，确认被审批。
2. 故意让任务尝试越界路径，确认被拦截。
3. 故意设置敏感上下文，确认 Privacy Ledger 有记录。
4. 故意超预算，确认有提示或降级策略。

### 20.4 打包 smoke 验证

1. 构建安装包。
2. 在干净环境安装。
3. 首次启动完成设置。
4. 完成一次真实任务闭环。
5. 检查卸载与升级路径无明显阻塞问题。
