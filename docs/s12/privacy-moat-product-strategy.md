# CodeMax 隐私护城河版产品方案（v1.0）

## 1. 方案目标

本方案用于定义 `CodeMax` 的首发产品方向呀。  
目标不是做一个泛泛的 AI 编程工具，而是做一版：

1. `能上线给其他人用`
2. `具备 Codex 核心能力`
3. `在隐私保护、个人专属画像、运行可控、交付可审计上明显强于 Codex 和 Cursor`
4. `强制纳入 D:\codemax\任务拆解.md 中 S12 及以后未完成工作`

这版产品的核心战略，不是“全面功能碾压所有竞品”，而是 `在一个足够硬、足够产品化、足够可感知的主赛道上打穿` 哦。

---

## 2. 最终产品定位

### 2.1 一句话定位

`CodeMax = 拥有 Codex 核心工作流，但在隐私保护、个人画像、运行可控和交付审计上明显更强的本地优先编程智能体产品`

### 2.2 与 Codex / Cursor 的竞争方式

我们不打“谁更像 IDE”，也不打“谁先接入最多模型”呀。  
我们要打的是：

1. `你能不能清楚知道 Agent 看了什么、发了什么、记住了什么`
2. `你能不能控制 Agent 这次到底允许做什么`
3. `你能不能拿到可信的交付证据，而不是只看到一堆输出`
4. `它能不能越用越像你，但又不黑箱`

### 2.3 最终产品表达

对外表达建议统一为：

`比通用 AI 编程工具更隐私、更可控、更懂你的本地编程智能体工作台`

---

## 3. 为什么选“隐私护城河版”

### 3.1 原因一：最适合当前代码基线

你现在的项目底座已经有这些东西呀：

1. worktree 隔离
2. 命令执行与日志
3. 审批与本地合并
4. Agent 状态机闭环
5. SQLite 结构化数据底座
6. memory 基础
7. `proof_packs`、`quality_gate_results`、`delivery_scores` 数据表

也就是说，你已经不是从零做 Agent 了，而是在做“从能跑到能上线”的产品收口哦。  
这时最值钱的方向，不是继续堆“会改代码”，而是把 `可信、可控、可解释` 做成主卖点呀。

### 3.2 原因二：最容易形成明显差异

Codex 和 Cursor 都强，但它们默认更像“强执行的通用编程智能体”呀。  
你可以切出一条更锐利的赛道：

1. 更强隐私可见性
2. 更强运行契约
3. 更强个人画像
4. 更强交付审计

这个赛道不是补一两个按钮就能追上的，它会形成真正的产品护城河哦。

### 3.3 原因三：更适合“先个人版，再预埋团队能力”

隐私、画像、记忆、预算、审计这些东西，本来就更适合先在个人版打透呀。  
等个人版站住之后，再往团队版延展就很顺了，比如：

1. 团队规则包
2. 团队 Hook
3. 团队审查标准
4. 团队私有部署

---

## 4. 首发产品战略结论

首发版本不追求“像完整 IDE”呀，而追求：

1. `Codex 核心能力不缺`
2. `隐私和控制能力明显更强`
3. `个人专属画像真正可见、可确认、可删除`
4. `交付结果真正可审查、可追责、可导出`

所以 `v1.0` 的目标不是“另一个 Codex”，而是：

`一个拥有 Codex 主链能力，但在隐私护城河上做出明确领先感的本地编程智能体产品`

## 5. Codex 核心能力基线清单

这一节定义“Codex 有的，我们首发也必须有”的能力底线呀。

### 5.1 任务主链基线

1. 选择本地仓库
2. 创建自然语言任务
3. 创建隔离 worktree
4. Agent 先规划 Todo，再执行
5. 命令执行、日志捕获、取消执行
6. 自动验证与失败后修复
7. 审批高风险行为
8. 展示最终 Diff
9. 展示测试/交付结果
10. 本地合并

### 5.2 任务工作台基线

1. 任务列表
2. 任务详情
3. Todo 看板
4. 运行时间线
5. 日志面板
6. 审批记录
7. 合并前检查

### 5.3 可用性基线

1. 重启后恢复任务
2. 多任务历史管理
3. 模型配置
4. 国际化基础
5. 桌面端可打包可安装

### 5.4 基线原则

这些能力不允许缺项呀。  
如果这些没有补齐，再强的隐私和画像能力也撑不起“能上线给别人用”的产品判断哦。

## 6. CodeMax 首发必须强于 Codex / Cursor 的维度

这里不是说全面碾压，而是要在用户能直接感知的维度形成明显领先呀。

### 6.1 维度一：隐私可见

用户必须知道：

1. 读了哪些文件
2. 哪些内容进了模型
3. 哪些内容被脱敏
4. 哪些内容被拦截
5. 用了哪家模型供应商

对应能力：

1. Privacy Ledger
2. 敏感路径/敏感内容拦截
3. 数据流向可视化摘要

### 6.2 维度二：运行可控

用户必须在任务开始前就知道：

1. 这次 Agent 能做什么
2. 不能做什么
3. 预算是多少
4. 验证命令是什么
5. Memory 会不会参与

对应能力：

1. Run Contract
2. Approval Center
3. Token Budget / Context Budgeter

### 6.3 维度三：个人画像可控

用户不能只感觉“它好像越来越懂我”呀，而要真的看到：

1. 它记住了什么
2. 为什么记住
3. 作用在哪些任务上
4. 我能不能删掉

对应能力：

1. Memory Cockpit
2. Preference Distiller
3. Personal Profile

### 6.4 维度四：交付可信

最终交付不能只是“代码改完了”，而必须是：

1. 有证据包
2. 有质量门禁
3. 有评分
4. 有风险说明
5. 有合并前统一审查页

对应能力：

1. Proof Pack
2. Quality Gate
3. Delivery Score
4. Delivery Review Page

## 15. 最终判断标准

如果首发版上线后，用户能明确感知到以下几点呀，我们就算真的立住了：

1. 它和 Codex 一样能跑完整任务链
2. 它比 Codex 更清楚地告诉我数据去了哪里
3. 它比 Cursor 更像我的专属工程搭子
4. 它给我的不是“代码输出”，而是“能审、能合、能复盘的交付结果”

如果这四点都成立，CodeMax 就不是普通的本地编程工具了，而是一个真正有产品护城河的亮点产品哦。
